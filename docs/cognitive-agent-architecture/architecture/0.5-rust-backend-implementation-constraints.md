# `0.5` Rust Backend Implementation Constraints

## Purpose

Turn the `0.5` canonical ontology into a backend-safe contract for a multiplatform Rust implementation.

The `0.5` object, registry, sync, and workflow model is only credible if the backend boundary rules are explicit enough to survive:

- multiple storage engines
- multiple target classes
- multiple runtime environments
- provider-feature gating
- mobile or embedded-adjacent builds later

This document defines those backend-facing constraints.

## Design Principle

Keep the canonical core small, typed, portable, and storage-agnostic.

Do not let convenience imports, storage-engine assumptions, platform handles, or provider-specific runtime details bleed into core domain contracts.

## Crate / Module Boundary Map

The backend should preserve explicit crate-like or module-like boundaries at minimum along these lines:

### `vel_core_types`

Owns:

- typed IDs and ID newtypes
- canonical durable envelope
- object-class taxonomy
- core object and registry enums
- relation enums
- schema/version metadata

### `vel_core_objects`

Owns:

- canonical content object definitions
- object validation
- canonical mutation logic
- object-level invariants

### `vel_core_registry`

Owns:

- `Module`, `Skill`, and `Tool` registry entities
- manifest-loading interfaces
- registry lookup/query
- registry reconciliation metadata

### `vel_core_relations`

Owns:

- typed directional relation model
- relation invariants
- relation traversal/query interfaces

### `vel_core_policy`

Owns:

- ownership rules
- confirmation posture
- policy evaluation
- policy explanation types
- grant and capability model contracts

### `vel_core_runtime`

Owns:

- `WriteIntent`
- run-record contracts
- approvals
- execution orchestration contracts
- scheduler/job interfaces

### `vel_core_sync`

Owns:

- `SyncLink`
- sync-state model
- reconciliation contracts
- source-summary derivation contracts

### Adapter crates

- `vel_adapters_todoist`
- `vel_adapters_google_calendar`

These own provider mapping logic but remain subordinate to the core object, policy, sync, and runtime contracts.

### `vel_storage`

Owns:

- repository traits
- transaction traits
- append/log interfaces
- embedded/backend-specific store implementations

### `vel_platform`

Owns platform seams such as:

- clock
- filesystem
- secret provider access
- target capability probing

## Storage-Agnostic Domain Rule

All durable logic must be defined against storage traits rather than SQL dialect specifics or one concrete persistence engine.

The core domain must not depend on:

- SQLx row shapes
- SQLite quirks
- Postgres-only features
- platform-specific key/value stores

Local embedded storage is the default posture, but the domain model must remain portable behind traits.

## Serialization And Wire Contract

### Canonical posture

- `serde` is the canonical serialization implementation strategy.
- JSON is the baseline interchange and debug format.
- Binary encodings may come later, but they are not contract truth in `0.5`.

### Required rules

- canonical objects must serialize deterministically
- runtime/control records must have versioned wire forms
- typed content IDs and registry IDs must have stable string encodings
- timestamps, enums, optional fields, and schema versions must use consistent cross-platform rules

### Compatibility rules to lock

- unknown-field handling must be explicit
- null-versus-omitted semantics must be explicit
- enum tagging strategy must be explicit
- forward/backward compatibility expectations must be explicit
- schema-version field location must be explicit

## ID Discipline

Use strong Rust newtypes for IDs inside the backend model, even when serialized as strings.

Examples:

- `TaskId`
- `EventId`
- `WorkflowId`
- `ThreadId`
- `ModuleId`
- `SkillId`
- `ToolId`
- `IntegrationAccountId`
- `SyncLinkId`
- `WriteIntentId`

This preserves ontology through the type system instead of relying on raw string discipline.

## Async And Ownership Discipline

Service traits used by runtime orchestration should be `Send + Sync` where target/runtime constraints allow it.

Public core contracts should:

- use owned-data structures
- avoid platform-specific handles
- avoid leaking runtime-local references/lifetimes across domain boundaries

The core must not assume a specific executor or thread model in its public contract shapes.

## Feature Gating

The backend should explicitly support high-level Rust feature segmentation such as:

- `todoist`
- `google-calendar`
- `sqlite`
- `postgres`
- `wasm`
- `native-secrets`
- `background-sync`
- `full-audit`
- `devtools`

### Required feature-gating rules

- the core object model compiles without provider adapters
- registry semantics compile without networked integrations
- read models degrade gracefully when provider features are disabled
- provider-specific code must not leak into core enums or contracts in ways that break stripped builds

## Target Capability Matrix

### Required across all targets

- stable clock access
- random or UUID generation
- canonical serialization/deserialization
- local persistence
- async task execution or equivalent emulation

### Optional by target

- background sync
- network reachability monitoring
- secure OS keychain
- push notifications
- filesystem watch
- multi-threaded executor

### Unsupported in core assumptions

- direct platform GUI handles
- platform-specific account APIs in the domain layer
- platform-specific secret APIs in core domain types

## Error Taxonomy

The backend should expose a canonical error model, not adapter-specific `anyhow` poetry.

Minimum error categories:

- validation error
- policy denied
- confirmation required
- conflict detected
- not found
- stale version
- sync failure
- provider auth failure
- provider rate limited
- storage failure
- unsupported capability
- migration failure

Layer contracts should define which categories they may emit.

## Concurrency And Versioning

### Required rules

- canonical objects carry revision/version
- updates use optimistic concurrency / compare-and-swap semantics or equivalent stale-write detection
- `SyncLink` and `WriteIntent` define stale-state behavior explicitly
- shared mapped fields define conflict behavior explicitly
- projections remain rebuildable and non-authoritative

## Bootstrap And Seeding

Bootstrap must be deterministic and idempotent.

### Required rules

- registry seeding precedes workflow seeding
- built-in workflow updates reconcile by stable identity and version
- local forks are never overwritten by bootstrap
- editable seeded workflows require drift markers instead of silent overwrite

## Secret Boundary

`IntegrationAccount` is canonical. Secrets are not.

Required rule set:

- canonical `IntegrationAccount` stores no raw secrets
- credentials are fetched through a `CredentialProvider` / `SecretStore`
- core domain logic sees auth/token availability state, not storage-mechanism details
- target-specific secret mechanisms remain behind platform/provider traits

## Query Abstraction

`object.query` and related queries must be defined as domain abstractions rather than storage-engine-specific ad hoc behavior.

The query model should define:

- filter AST or query-struct contract
- pagination contract
- sort semantics
- relation traversal semantics
- archive/tombstone visibility flags
- projection/read-model query behavior

## Scheduler And Runtime Abstraction

The backend should define platform-neutral runtime roles for:

- enqueueing sync jobs
- processing write intents
- workflow run orchestration
- retry/backoff
- cancellation
- idempotency keys
- crash recovery/resume

Do not let adapters or workflows smuggle executor-specific task spawning in as architecture.

## Projection / Read-Model Separation

Projection/read-model layers must remain distinct from canonical truth layers.

Rules:

- canonical object crates define truth
- projection/read-model crates define derivations
- cached summaries such as `source_summary` are convenient, not authoritative
- rebuild procedures must exist

## Testability Requirements

The backend contract should support:

- in-memory store implementations
- deterministic fixture seeding
- fake clock implementations
- fake provider adapters
- replayable migration tests
- policy explanation golden tests
- sync reconciliation scenario tests

The `0.5` proving flows should be runnable against:

- in-memory test backend
- local embedded backend
- real adapter integration tests where applicable

## `no_std` / Wasm Stance

`0.5` does not need to promise full `no_std`, but it should preserve a domain core that avoids unnecessary OS coupling where reasonable.

Recommended stance:

- core types and policy logic avoid unnecessary OS-specific dependencies
- storage/network adapters remain target-specific
- runtime execution varies by target behind traits

## Summary

The `0.5` backend should answer four questions cleanly:

- what compiles where
- what persists through what
- what runs through what
- what is allowed to depend on what

That is the difference between a clean ontology and a clean system.
