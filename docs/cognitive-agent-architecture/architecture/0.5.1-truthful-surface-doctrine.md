# `0.5.1` Truthful Surface Doctrine

## Status

Authority doc for milestone `v0.5.1`.

## Purpose

`v0.5.1` is a truth-alignment milestone. Its job is to make client surfaces conform to the frozen `0.5` backend rather than renegotiate backend law or widen product scope.

## Invariant

- clients conform to canonical backend contracts
- backend contracts are immutable during `v0.5.1` except for provable backend bugs
- no schema negotiation
- no client-invented semantic truth

## Truthful Surface Criteria

A client surface is truthful only if all of the following are true.

### Data

- only canonical read models are rendered
- direct client consumption of legacy DTO-shaped truth is forbidden
- client-side data shaping may format or group, but may not create semantic meaning

### Mutation

- all mutations must flow through canonical `WriteIntent`
- direct state mutation is forbidden
- client-side mutation shortcuts are forbidden

### Derivation

Allowed:

- formatting
- grouping for readability
- presentational ordering that is explicitly delivered by backend truth

Forbidden:

- client-side prioritization
- inferred urgency
- semantic reclassification
- local workflow simulation
- local provider-specific normalization of canonical truth

### Optimistic UI

Allowed:

- loading states
- pending indicators
- local drafts

Forbidden:

- optimistic semantic truth
- optimistic task/calendar state that pretends backend reconciliation already happened

### Failure Posture

- development and test must fail loudly on contract drift
- production may degrade only into explicit error/degraded states
- silent fallback is forbidden

## Transport Law

- one typed query layer
- one typed mutation layer
- no direct fetches outside those layers
- no client-side normalization of backend truth outside those layers

## Surface Model

`v0.5.1` defines exactly three client surfaces:

- `Now` — temporal and operational surface
- `Threads` — contextual and interaction surface
- `System` — structural, object, capability, and configuration surface

Explicit exclusions:

- `Inbox` is not a first-class surface
- `Settings` is not a first-class surface
- no additional top-level surfaces are allowed in this milestone

## Surface-Specific Rules

### Now

- only canonical inputs are allowed
- direct triage mutations are allowed only through canonical `WriteIntent`
- no client-side prioritization, inferred urgency, or hidden ranking rules

### Threads

- contract rebinding only
- manual workflow invocation may exist only when object-scoped
- thread-scoped invocation without a bound object is forbidden

### System

- one top-level route: `/system`
- internally sectioned, not split into mini-app routes
- object browsing, invocation, and configuration only
- no builders, editors, or workflow simulation

## Cleanup Law

`v0.5.1` is not complete until all of the following are true:

- deprecated routes are deleted or explicitly quarantined with named rationale
- no direct fetches exist outside the canonical transport layer
- no legacy DTO normalization remains in views
- no client-side semantic truth remains
- all mutations go through canonical paths

## Non-Goals

- backend ontology changes
- new providers
- Apple implementation
- framework migration
- broad UI redesign
- scheduler/trigger automation
- workflow-builder product work
