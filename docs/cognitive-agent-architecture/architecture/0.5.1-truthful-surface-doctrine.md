# `0.5.1` Truthful Surface Doctrine

## Status

Authority doc for milestone `v0.5.1`.

## Purpose

`v0.5.1` is a truth-alignment milestone. Its job is to make client surfaces conform to the frozen `0.5` backend rather than renegotiate backend law or widen product scope.

## Invariant

- clients conform to canonical backend contracts
- backend contracts are immutable during `v0.5.1` except for provable backend bugs
- no schema negotiation
- no client-invented semantic truth

## Truthful Surface Criteria

A client surface is truthful only if all of the following are true.

### Data

- only canonical read models are rendered
- direct client consumption of legacy DTO-shaped truth is forbidden
- client-side data shaping may format or group, but may not create semantic meaning

### Mutation

- all mutations must flow through canonical `WriteIntent`
- direct state mutation is forbidden
- client-side mutation shortcuts are forbidden

### Derivation

Allowed:

- formatting
- grouping for readability
- presentational ordering that is explicitly delivered by backend truth
- adjacent canonical sections within a surface when the backend does not provide a merged canonical model

Forbidden:

- client-side prioritization
- inferred urgency
- semantic reclassification
- cross-section ordering that implies shared priority where no canonical merged model exists
- local workflow simulation
- local provider-specific normalization of canonical truth

### Optimistic UI

Allowed:

- loading states
- pending indicators
- local drafts

Forbidden:

- optimistic semantic truth
- optimistic task/calendar state that pretends backend reconciliation already happened

### Failure Posture

- development and test must fail loudly on contract drift
- production may degrade only into explicit error/degraded states
- silent fallback is forbidden
- last-known-good data may be shown only when its canonical provenance and staleness are explicit
- otherwise the affected component must fail explicitly rather than render guessed or unlabeled stale truth

## Transport Law

- one typed query layer
- one typed mutation layer
- no direct fetches outside those layers
- no client-side normalization of backend truth outside those layers

## Surface Model

`v0.5.1` defines exactly three client surfaces:

- `Now` — temporal and operational surface
- `Threads` — contextual and interaction surface
- `System` — structural, object, capability, and configuration surface

Explicit exclusions:

- `Inbox` is not a first-class surface
- `Settings` is not a first-class surface
- no additional top-level surfaces are allowed in this milestone

## Surface-Specific Rules

### Now

- only canonical inputs are allowed
- tasks and calendar commitments may appear only as adjacent canonical sections, not one merged ranked feed
- no synthetic cross-section ordering is allowed unless the backend later ships an explicit merged canonical model
- direct triage mutations are allowed only through canonical `WriteIntent`
- in-scope triage mutations are `complete`, `reopen`, `defer` or `snooze` via explicit due/date mutation, state change, and quick due/date move only when the backend action is already canonical and stable
- post-mutation visibility is determined by backend reconciliation and read-model truth, not client-kept ghost state
- no client-side prioritization, inferred urgency, or hidden ranking rules

### Threads

- contract rebinding only
- manual workflow invocation may exist only when object-scoped
- invocation requires exactly one bound canonical object with a stable object ID and supported workflow binding
- threads with zero bound objects may guide users to attach or create an object first, but may not invoke anything
- multi-object invocation is forbidden in `v0.5.1`

### System

- one top-level route: `/system`
- internally sectioned, not split into mini-app routes
- exact first-pass sections are:
  - `Domain`: `People`, `Calendar`, `Knowledge`
  - `Capabilities`: `Tools`, `Workflows`, `Templates`
  - `Configuration`: `Modules`, `Integrations`, `Accounts`, `Scopes`
- object browsing, invocation, and configuration only
- read-only browsing is the default posture across all sections
- invocation is allowed only in `Tools`, `Workflows`, and already-stable canonical object actions
- `Templates` are list/read-only unless a real canonical apply or invoke action already exists
- `Knowledge` remains thin browse-plus-stable-action territory only
- `Modules`, `Integrations`, `Accounts`, and `Scopes` may render only these named canonical single-step actions:
  - `Modules`: `activate`, `deactivate`
  - `Integrations`: `connect`, `disconnect`, and `refresh` only when `refresh` already exists as an unambiguous canonical backend action
  - `Accounts`: `reauthorize`, `disconnect`
  - `Scopes`: `enable`, `disable` only when those actions already exist canonically
- no additional actions may be rendered even if technically available
- composite actions are forbidden
- inferred actions from state are forbidden
- client-side interpretation of capability is forbidden
- ambiguous actions are excluded unless their backend contract is already explicit and unambiguous
- no builders, editors, or workflow simulation

## Explain Posture

- explain UI renders a curated summary first
- raw backend payload may be shown only as an explicit expandable detail view
- client-inferred explanation is forbidden

## Cleanup Law

`v0.5.1` is not complete until all of the following are true:

- deprecated routes are deleted or explicitly quarantined with named rationale
- no direct fetches exist outside the canonical transport layer
- no legacy DTO normalization remains in views
- no client-side semantic truth remains
- all mutations go through canonical paths

## Non-Goals

- backend ontology changes
- new providers
- Apple implementation
- framework migration
- broad UI redesign
- scheduler/trigger automation
- workflow-builder product work
