# `0.5.2` Operator Surface Doctrine

## Purpose

`v0.5.2` is the ideal-state operator-surface milestone.

It is not a backend rewrite, not a framework migration, and not a broad provider-expansion line. The backend law frozen in `0.5` and the truthful client boundary frozen in `0.5.1` remain authoritative. `0.5.2` exists to make the operator experience feel intentional, fast, and structurally coherent on top of those frozen seams.

## Surface Model

The product still has exactly three first-class surfaces:

- `Now` — temporal and operational
- `Threads` — contextual and interactional
- `System` — structural and configurational

`Inbox` remains absorbed into `Now`. `Settings` remains replaced by `System`. No new top-level surfaces are introduced in this milestone.

## Core Law

- canonical backend truth remains authoritative
- no backend schema negotiation except for provable bugs
- web is the active embodiment target
- Apple remains out of implementation scope and receives parity/handoff updates only
- layout, density, and interaction quality may change substantially
- client-side semantic invention remains forbidden

## Ideal-State Standard

An operator surface in `0.5.2` should be:

- direct enough to operate without hunting
- dense enough to feel useful at first glance
- fast enough that mutation and navigation do not feel delayed
- structurally legible enough that the operator can tell where truth lives
- thin enough that meaning still comes from backend models rather than local improvisation

## Surface-Specific Rules

### `Now`

- remains one surface, but tasks and calendar commitments stay in adjacent canonical sections rather than a synthetic merged priority feed
- may offer direct triage actions only through canonical mutation paths
- may not invent urgency, ranking, or scheduling semantics locally
- should optimize for immediate orientation, execution, and current-day intervention

### `Threads`

- remains grounded in canonical objects and bounded interaction context
- workflow and tool affordances stay narrow and explicit
- invocation eligibility is shown truthfully; the client does not simulate orchestration
- the surface should make provenance, object grounding, and actionable next steps easier to read

### `System`

- remains a single top-level route with internal sections
- is read-first, invoke-narrowly, and mutate-structurally only through already canonical actions
- should expose structure, capability state, and configuration truth without turning into an admin sprawl surface

## Performance Law

`0.5.2` is allowed to change UI substantially, but it is not allowed to accept obvious sluggishness as the cost of truthfulness.

This milestone must reduce visible latency in the active operator path by:

- tightening route-level loading behavior
- reducing oversized inspect/config payload dependency where possible without reopening backend law
- removing unnecessary client churn and refetch patterns in active surfaces
- proving browser-visible reads and mutations on the shipped surface

## Explicit Non-Goals

- no new providers
- no workflow builder
- no trigger or scheduler automation widening
- no Apple implementation
- no backend ontology renegotiation
- no framework migration

## Verification Posture

Claims about operator-visible behavior must be proven in the browser.

Claims about shipped surface structure must be visible in artifacts and code.

Claims about mutation correctness must be proven with backend state, logs, or execution-backed browser evidence as appropriate.
