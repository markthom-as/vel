# `0.5.2` Operator Surface Doctrine

## Purpose

`v0.5.2` is the ideal-state operator-surface milestone.

It is not a backend rewrite, not a framework migration, and not a broad provider-expansion line. The backend law frozen in `0.5` and the truthful client boundary frozen in `0.5.1` remain authoritative. `0.5.2` exists to make the operator experience feel intentional, fast, and structurally coherent on top of those frozen seams.

## Surface Model

The product still has exactly three first-class surfaces:

- `Now` — temporal and operational
- `Threads` — contextual and interactional
- `System` — structural and configurational

`Inbox` remains absorbed into `Now`. `Settings` remains replaced by `System`. No new top-level surfaces are introduced in this milestone.

## Shared Frame Law

- all three surfaces share one page-frame anatomy
- `Now` may receive stronger emphasis, but not structural deviation
- the old global right-side info rail is removed from shell law
- secondary inspection returns only as per-surface drawer or contextual expansion
- navigation keeps icon plus label on desktop and compact labels on narrow screens
- icon-only navigation states are forbidden

## Core Law

- canonical backend truth remains authoritative
- no backend schema negotiation except for provable bugs
- no new backend endpoints unless pre-declared in the milestone packet
- web is the active embodiment target
- Apple remains out of implementation scope and receives parity/handoff updates only
- layout, density, and interaction quality may change substantially
- client-side semantic invention remains forbidden

## Ideal-State Standard

An operator surface in `0.5.2` should be:

- direct enough to operate without hunting
- dense enough to feel useful at first glance
- fast enough that mutation and navigation do not feel delayed
- structurally legible enough that the operator can tell where truth lives
- thin enough that meaning still comes from backend models rather than local improvisation

## Surface-Specific Rules

### `Now`

- general `Now` layout may not change silently during implementation
- layout and visual design for `Now` must be defined together before embodiment work lands
- any material change to `Now` layout requires explicit operator approval first
- has a fixed first-pass priority gradient:
  - `Focus`
  - `Commitments`
  - `Calendar`
  - `Triage`
- remains one surface, but tasks and calendar commitments stay in adjacent canonical sections rather than a synthetic merged priority feed
- `Focus` is the current execution locus, not just another list
- `Focus` may contain one dominant item and at most one clearly subordinate next-up slot
- may offer direct triage actions only through canonical mutation paths
- may not invent urgency, ranking, or scheduling semantics locally
- calendar scope is today-first with only near-horizon bleed
- capture remains always available but visually subordinate and zero-friction
- `Commitments` split into `Overdue` and `Due soon` when both exist
- `Triage` shows compressed project/source cues by default rather than hiding identity on hover
- should optimize for immediate orientation, execution, and current-day intervention

### `Threads`

- remains grounded in canonical objects and bounded interaction context
- bound object state is the primary unit of emphasis, not message chronology
- default layout is object summary plus active thread when a bound object exists
- fallback is thread list plus active thread only when no bound object exists
- workflow and tool affordances stay narrow and explicit
- invocation eligibility is shown truthfully; the client does not simulate orchestration
- provenance defaults to compressed inline signal with expandable deep detail
- deep provenance belongs in a dedicated panel or drawer, not inline expansion
- the surface should make provenance, object grounding, and actionable next steps easier to read

### `System`

- remains a single top-level route with internal sections
- uses three visibly distinct groups:
  - `Domain`
  - `Capabilities`
  - `Configuration`
- is read-first, invoke-narrowly, and mutate-structurally only through already canonical actions
- prefers split-pane detail on desktop and stacked drill-in on narrow screens
- uses sidebar hierarchy on desktop and grouped list navigation on narrow screens
- visual tone stays mostly monochrome with sparse meaningful accents only
- should expose structure, capability state, and configuration truth without turning into an admin sprawl surface

## Performance Law

`0.5.2` is allowed to change UI substantially, but it is not allowed to accept obvious sluggishness as the cost of truthfulness.

This milestone must reduce visible latency in the active operator path by:

- making surface loads immediately recognizable with stable skeletons
- reconciling mutations locally first and globally second
- avoiding full-surface invalidation except in catastrophic cases
- keeping interaction latency below the attention threshold in operator feel
- preserving predictive stability so the UI does not jump after actions
- tightening route-level loading behavior
- reducing oversized inspect/config payload dependency where possible without reopening backend law
- removing unnecessary client churn and refetch patterns in active surfaces
- proving browser-visible reads and mutations on the shipped surface

Client-side caching and prefetch are allowed only when they do not create temporal ambiguity, stale authority illusions, or post-hydration reordering surprises.

## Visual Direction

A strong visual reset is allowed in `0.5.2`.

That allowance does not authorize unilateral `Now` layout churn. `Now` layout and design must be locked together before implementation changes that materially alter the surface structure.

That reset must preserve:

- hierarchy clarity
- state legibility
- action-affordance precision

The target is an instrument panel, not a landing page.

## Explicit Non-Goals

- no new providers
- no workflow builder
- no trigger or scheduler automation widening
- no Apple implementation
- no backend ontology renegotiation
- no framework migration

## Verification Posture

Claims about operator-visible behavior must be proven in the browser.

Claims about shipped surface structure must be visible in artifacts and code.

Claims about mutation correctness must be proven with backend state, logs, or execution-backed browser evidence as appropriate.

Per-surface proof is required, and one full operator-loop proof is also required.

Apple parity in this line means the same doctrine with platform-native expression, not pixel parity with web.

## Baseline Preservation

Before shell or surface embodiment changes begin, the current UI baseline must be snapshotted in a recoverable form. Recoverability may be satisfied by a named commit/tag reference plus a checked-in baseline note pointing back to the pre-change UI state.
