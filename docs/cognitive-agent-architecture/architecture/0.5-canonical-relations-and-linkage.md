# Purpose

Define how canonical objects relate to one another and to external provider state in milestone `0.5`.

# Relation Model

Relations in `0.5` are:

- strongly typed
- directional
- explicit
- storage-agnostic
- optimistic-concurrency-aware

Relations are not freeform labels or untyped adjacency blobs.

Examples:

- `task -> project`
- `task -> parent_task`
- `event -> calendar`
- `event -> attendee`
- `workflow -> tool`
- `object -> sync_link`

Each relation record must at minimum identify:

- source object ID
- relation type
- target object ID
- direction
- created/updated revision metadata

# IntegrationAccount

`IntegrationAccount` is a first-class canonical content object.

It identifies:

- provider family
- account identity in Vel
- scope/grant posture
- activation state
- sync posture
- account-level policy posture

It does not store credentials. Credentials remain outside the canonical content substrate and must be fetched through explicit secure backend traits.

# SyncLink

`SyncLink` is a first-class canonical content object.

It is the canonical linkage record between a Vel object and a provider-side object or target. It exists to keep linkage truth externalized instead of smuggling it into per-object JSON.

At minimum, a `SyncLink` carries:

- `id`
- `provider`
- `integration_account_id`
- `object_id`
- `remote_id`
- `remote_type`
- `state`
- `authority_mode`
- `remote_version`
- `linked_at`
- `last_seen_at`

# source_summary Boundary

`source_summary` may report linkage facts compactly, but it is always derived from `SyncLink` plus relations.

Allowed uses:

- compact object summaries
- explain/debug output
- query ergonomics

Forbidden use:

- acting as the source of truth for linkage resolution

# Cardinality Rules

`SyncLink` supports history and relinking.

Allowed:

- many historical links over time
- multiple links from one canonical object across providers
- multiple links from one canonical object across accounts

Invariants:

- at most one active authoritative link per live canonical-object/provider/account/remote target pairing
- at most one active link per `(provider, integration_account_id, remote_id, remote_type)` tuple
- superseded links remain historical instead of being deleted

# Lifecycle States

`SyncLink` lifecycle vocabulary in `0.5`:

- `active`
- `superseded`
- `broken`
- `deleted_upstream`
- `reconciled`
- `conflicted`

These states are canonical enough to drive explainability, reconciliation, and cutover logic.

# Tombstone And Restore Posture

Upstream deletes produce local tombstones immediately.

Default query posture:

- tombstones hidden from ordinary query paths
- explicitly queryable for reconciliation, audit, and explain

Restore rule:

- revive the existing canonical object when provider/account/remote identity confidence is strong
- preserve historical tombstone interval and linkage lineage
- create a new canonical object only when prior identity cannot be safely re-established

# Concurrency Posture

Linkage operations must be optimistic-concurrency-aware.

Implications:

- sync-link writes use compare-and-swap or equivalent version semantics
- stale linkage updates do not silently overwrite newer linkage state
- relation and linkage projections remain rebuildable

# Rust Backend Posture

The relation and linkage model must remain:

- storage-agnostic
- serde-compatible
- versioned
- newtype-friendly
- feature-gated at provider/storage boundaries

No canonical linkage contract may depend on SQL-specific joins, provider SDK handles, or platform-specific credential APIs.

# Summary

`0.5` keeps linkage law explicit:

- first-class `IntegrationAccount`
- first-class `SyncLink`
- directional typed relations
- derived `source_summary`
- historical plus active linkage
- explicit active, superseded, broken, deleted_upstream, reconciled, and conflicted state
