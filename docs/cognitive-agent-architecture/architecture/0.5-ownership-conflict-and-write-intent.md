# Purpose

Define how `0.5` decides who owns fields, how conflicts are represented, and how outward writes cross from canonical mutation requests into governed provider-side execution.

# Ownership Model

Field ownership in `0.5` uses:

- static defaults
- dynamic overlays

Static defaults express normal provider/core responsibility.

Dynamic overlays express local policy, account-level posture, migration state, or object-level exceptions without rewriting the whole ontology.

# Ownership Classes

At minimum, `0.5` distinguishes:

- `source-owned`
- `shared`
- `Vel-only`

`source-owned` fields follow provider authority on conflict.

`shared` fields require explicit merge or policy-mediated resolution.

`Vel-only` fields are canonical fields that do not round-trip unless an explicit adapter mapping says otherwise.

# Conflict And Staleness

The contract must distinguish:

- stale version
- ownership conflict
- pending reconciliation
- provider divergence
- tombstone/write race

Stale version is not the same thing as semantic conflict.

# Tombstone Posture

Tombstones are explicit canonical state, not disappearance.

Rules:

- upstream deletes create tombstones immediately
- tombstones stay hidden from default query paths
- tombstones remain available for explain, audit, and reconciliation
- restore/reappearance should prefer reviving existing canonical identity where confidence is strong

# Pending Write Intent Rule

Local canonical edits that may require outward propagation become a pending write intent first.

This is true even when policy is permissive and execution is effectively immediate.

# WriteIntent

`WriteIntent` is one provider-agnostic runtime record type in `0.5`.

It is not split into separate provider-specific ontologies up front.

Required lifecycle vocabulary:

- `proposed`
- `approved`
- `executing`
- `succeeded`
- `failed`
- `cancelled`

`WriteIntent` is the formal bridge between:

- canonical mutation request
- ownership evaluation
- policy evaluation
- confirmation
- outward provider execution

# WriteIntent Fields

At minimum, a `WriteIntent` carries:

- `id`
- `action_name`
- `target_object_refs`
- `provider`
- `integration_account_id`
- `requested_change`
- `ownership_evaluation`
- `policy_evaluation`
- `confirmation_mode`
- `status`
- `downstream_operation_refs`

# Read-Only Precedence

Read-only controls may deny write execution at:

- workspace level
- module level
- integration account level

More restrictive policy wins.

This precedence applies before outward execution begins.

# Explainability

Ownership and conflict handling must remain explainable.

At minimum, explain paths should identify:

- field ownership basis
- overlay influence
- whether the result is source-favored
- whether a pending write intent was created
- whether confirmation was required

# Error Taxonomy References

This contract expects membrane-safe error categories such as:

- `validation error`
- `policy denied`
- `confirmation required`
- `stale version`
- `storage failure`

# Non-Goals

This contract does not:

- define concrete adapter payload transforms
- treat provider comments, jobs, or audits as canonical content objects
- allow direct provider mutation without `WriteIntent`

# Summary

`0.5` keeps outward mutation law explicit:

- static defaults plus dynamic overlays
- `source-owned`, `shared`, and `Vel-only` field classes
- tombstone-first deletion posture
- pending write intent before external mutation
- one provider-agnostic `WriteIntent` lifecycle
