# Purpose

Define the lawful backend membrane for `0.5` so that every read, mutation, sync write, and workflow-mediated operation passes through one stable contract.

# Action Namespace

The `0.5` membrane is grounded in object-generic actions first.

Required generic actions:

- `object.get`
- `object.query`
- `object.create`
- `object.update`
- `object.delete`
- `object.link`
- `object.explain`

Specialized aliases may exist on top of these for ergonomics or bounded domain semantics, but they must not become separate sovereign policy systems.

# Action Contract

Each action must define:

- action name
- target object refs
- input payload schema
- output payload schema
- policy hooks
- audit hooks
- confirmation mode
- typed error surface

This keeps the membrane explicit instead of reducing it to “whatever service method happened to run.”

# Policy Layers

Policy evaluation in `0.5` is layered and explainable.

At minimum, policy may operate at:

- workspace
- module
- integration account
- object
- action

These layers must be composable and inspectable rather than implicit.

# Read-Only Posture

Read-only controls must exist at:

- workspace scope
- module scope
- integration account scope

An enabled module or connected provider account is not equivalent to free mutation authority.

# Confirmation Modes

The confirmation vocabulary in `0.5` is:

- `auto`
- `ask`
- `ask_if_destructive`
- `ask_if_cross_source`
- `ask_if_external_write`
- `deny`

These modes govern action execution posture. They do not replace policy; they are one output of policy and action evaluation.

# Explainability Requirements

The following are first-class contract requirements:

- `policy.explain`
- `object.explain`

`policy.explain` must surface:

- which policy layers applied
- the effective confirmation mode
- why an action was allowed, narrowed, or denied

`object.explain` must surface:

- source_summary basis
- linkage and ownership context
- policy-relevant status where useful

# Runtime And Trait Posture

The membrane contract must remain backend-safe:

- storage-agnostic
- trait-oriented
- serde-compatible
- versioned
- feature-gated at provider and storage boundaries

Runtime services that enforce the membrane should be compatible with typed Rust newtype IDs and explicit query/scheduler/secret seams rather than platform-specific shortcuts.

# Error Taxonomy At The Membrane Boundary

The membrane should expose stable categories such as:

- `validation error`
- `policy denied`
- `confirmation required`
- `not found`
- `stale version`
- `conflict detected`
- `storage failure`
- `unsupported capability`

These are backend-safe categories, not adapter-specific prose.

# External Write Rule

External writes never bypass the membrane.

If an action leads to provider-side mutation, the membrane must produce or route through a `WriteIntent` runtime record even when policy permits near-immediate execution. See [0.5-ownership-conflict-and-write-intent.md](/home/jove/code/vel/docs/cognitive-agent-architecture/architecture/0.5-ownership-conflict-and-write-intent.md).

# Non-Goals

This contract does not:

- define provider-specific API operations
- define storage-engine-specific implementation
- grant raw-tool access to skills
- collapse workflow/runtime behavior into provider adapters

# Summary

`0.5` has one membrane:

- generic actions first
- layered policy
- explicit read-only posture
- required `policy.explain`
- required `object.explain`
- typed confirmation and error surfaces
- no bypass around `WriteIntent` for external writes
