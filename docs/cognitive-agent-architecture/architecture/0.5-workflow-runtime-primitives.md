# Purpose

Define the minimum lawful workflow contract for `0.5` so seeded and user-authored workflows share one storage and runtime model without reopening ontology questions in later phases.

# Workflow As Canonical Content Object

`Workflow` is a canonical content object.

All workflows are stored in canonical storage whether they are:

- seeded
- imported
- user-authored

There is no manifest-only workflow class in `0.5`.

Built-in workflows may originate from manifests, but bootstrap materializes them into canonical workflow objects so policy, audit, explain, and references use one stable substrate.

# Provenance

Required provenance vocabulary:

- `seeded`
- `user-authored`
- `imported`

Practical origin values may also include `system` when the source is bundled product logic, but the important rule is that seeded and user-authored workflows remain the same object family.

# Mutability

Workflow mutability is explicit:

- `immutable`
- `forkable`
- `editable`

Seeded workflows default to fork-before-modify.

That means:

- most seeded workflows are `immutable` or `forkable`
- `editable` seeded workflows are rare and reserved for clearly local or operator-owned configuration behavior
- fork-before-modify is the default unless the workflow is explicitly marked `editable`

Editable seeded workflows must carry explicit drift and reconciliation metadata so local changes are not silently overwritten.

Recommended fields include:

- `seed_version`
- `local_modified_at`
- `local_modified_by`
- `upstream_update_available`
- `reconciliation_state`

# Canonical Workflow IDs

Workflows use typed opaque content IDs such as:

- `workflow_...`

They do not use human-readable semantic registry IDs as their primary identity.

# Seeded Workflow Reconciliation

Workflow bootstrap must remain deterministic bootstrap and idempotent.

Rules:

- bundled workflow manifests seed canonical workflow objects on bootstrap
- rerunning bootstrap updates or preserves existing workflow objects by stable identity and provenance
- local forks are never overwritten
- editable seeded workflows surface drift rather than silently reconciling it away

# Workflow Runtime Primitive Contract

`0.5` defines a deliberately small workflow runtime primitive set.

Minimal step taxonomy:

- `action`
- `skill`
- `approval`
- `sync`
- `condition`

The taxonomy stays intentionally narrow. It does not authorize generic hooks, broad orchestration features, or provider-specific side doors.

# Grant Envelope

Every workflow invocation operates under an explicit grant envelope.

The grant envelope defines:

- caller scope
- workflow-declared requests
- module and workspace policy constraints
- integration-account posture where relevant
- resulting effective grants

Workflows do not inherit ambient authority simply because they exist.

# Skills And Raw Tools

skills do not call raw tools

Workflow steps may invoke `skill` primitives, but those skill invocations remain mediated through the action membrane, effective grant envelope, confirmation rules, and audit path.

Direct raw-tool execution is not the default workflow seam for `0.5`.

# Hooks

Hooks are reserved but deferred.

The workflow contract must mention hooks so the future seam is explicit, but executable hooks are not part of the required `0.5` primitive runtime surface.

Do not widen this phase into general hook execution semantics.

# Approval And Sync Steps

`approval` steps are declarative workflow pauses that request or await confirmation through the same runtime and policy substrate.

`sync` steps exist for bounded synchronization or reconciliation triggers, but they remain membrane-governed and auditable rather than provider-side shortcuts.

# Runtime Boundaries

The workflow runtime contract must remain backend-safe:

- storage-agnostic
- serde-compatible
- trait-oriented
- compatible with feature gating
- compatible with deterministic bootstrap and idempotent seeded workflow loading

The runtime does not reopen module loading logic; it consumes the registry contract provided by `ManifestSource`, `RegistryLoader`, and `RegistryReconciler`.

# Non-Goals

This contract does not:

- define scheduler or trigger automation
- define executable hooks
- define provider adapter APIs
- allow workflows or skills to bypass the membrane

# Summary

`0.5` workflows are canonical content objects with one storage model for seeded and user-authored records, explicit `immutable` / `forkable` / `editable` mutability, fork-before-modify defaults, one small step taxonomy, explicit grant envelope semantics, and a hard rule that skills do not call raw tools.
