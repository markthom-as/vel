# `0.5` Required Backend Traits And Capability Matrix

## Purpose

Name the backend interfaces and cross-target assumptions required to implement `0.5` cleanly in Rust.

This document is not code. It is the contract map that prevents later phases from inventing storage, runtime, secret, and scheduling seams ad hoc.

## Required Backend Traits

### Storage and transactions

- `ObjectStore`
- `RegistryStore`
- `RelationStore`
- `SyncLinkStore`
- `RuntimeStore`
- `AuditStore`
- `ProjectionStore`
- `TransactionManager`

### Policy and registry

- `PolicyEvaluator`
- `RegistryLoader`
- `ManifestSource`
- `RegistryReconciler`

### Platform and secrets

- `Clock`
- `CredentialProvider`
- `SecretStore`

### Runtime orchestration

- `JobScheduler`
- `RuntimeExecutor`
- `IntentProcessor`

## Capability Matrix

| Local persistence | yes | no | no |

| Background sync | no | yes | no |

| Platform secret APIs in domain types | no | no | yes |

## Runtime Discipline

Interfaces used by orchestration should be runtime-portable:

- async-safe where appropriate
- owned-data contracts
- no platform-specific handle leakage
- executor-neutral public boundaries

This matrix assumes feature gating at provider, storage, and runtime edges rather than platform-specific branching inside core contracts.

## Feature-Gating Expectations

Provider and platform features should be segmentable.

Illustrative features:

- `todoist`
- `google-calendar`
- `sqlite`
- `postgres`
- `wasm`
- `native-secrets`
- `background-sync`
- `full-audit`
- `devtools`

Rules:

- core object model compiles without providers
- registry semantics compile without networked integrations
- read models degrade when feature-disabled
- provider-specific logic stays outside core contracts

## Testability Contract

The backend traits must support:

- in-memory implementations
- deterministic seed fixtures
- fake clock
- fake provider adapters
- migration replay tests
- policy explanation golden tests
- sync-reconciliation scenarios

The same trait set should support fake adapter tests early and real smoke verification later without changing the core contracts.

## Summary

These traits and capability expectations are the minimum backend scaffolding required to keep `0.5` portable, testable, and enforceable across target classes.
