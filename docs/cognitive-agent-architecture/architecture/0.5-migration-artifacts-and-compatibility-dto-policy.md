# Purpose

Define the lawful migration artifact shape, compatibility DTO layer limits, and proving flows required to cut `0.5` in without undocumented import or compatibility behavior.

# Migration Artifact

A migration artifact is the durable record of old-to-new transformation input, validation, and replay posture.

Required migration artifact concepts:

- artifact identity
- version
- source system description
- snapshot ref
- validation result
- transformation result
- replay metadata
- idempotence expectations

Migration artifacts exist so migration artifact import can be replayed, inspected, and verified instead of being treated as one-off scripts with amnesia.

# Artifact Contract

Each migration artifact should capture:

- source snapshot identity and snapshot ref
- source schema version or legacy shape hint
- canonical target schema version
- validation issues and warnings
- transformed object counts and relation counts
- tombstone handling summary
- replay attempts and replay status

The machine-readable draft lives at [0.5-migration-artifact.schema.json](/home/jove/code/vel/config/schemas/0.5-migration-artifact.schema.json).

# Replay And Idempotence

Replay must be explicit and idempotent.

Rules:

- rerunning the same artifact with the same input should not duplicate canonical objects
- replay must surface whether the result was unchanged, updated, skipped, or failed
- replay must preserve auditability of prior attempts
- replay must interact safely with deterministic bootstrap and stable IDs

Idempotence is not optional because the cutover path has to survive retries, interrupted runs, and real migration debugging.

# Validation

Validation must remain first-class.

At minimum, migration artifacts must expose:

- `validation`
- error count
- warning count
- blocking versus non-blocking issues
- confidence or basis where inference is involved

Migration cannot be treated as successful merely because a script completed.

# Compatibility DTO Layer

Compatibility DTO layer support is temporary and bounded.

The Compatibility DTO layer may exist only to:

- bridge old transport or persistence shapes during cutover
- isolate legacy endpoints that cannot be removed in the same slice
- provide narrowly-scoped temporary mapping while canonical surfaces are taking over

The Compatibility DTO layer must not become a permanent alternate truth path.

# Removal Criteria

Compatibility DTO layer removal criteria are explicit.

Removal criteria:

- no legacy write path remains active
- canonical DTOs or domain records own the live write path
- temporary legacy read paths are isolated and documented
- milestone evidence identifies any quarantined survivor endpoints
- follow-up removal work is ticketed if a read-only survivor remains

If any legacy path still handles writes at Phase 65 closeout, the milestone fails.

# Proving Flows

The proving flows for `0.5` are mandatory, not aspirational.

Must-pass milestone flows include:

- Todoist backlog import
- Google Calendar bounded import
- one Todoist write through `WriteIntent`
- one Google Calendar write through `WriteIntent`
- one denial path per adapter
- one dry-run path
- one workflow over canonical task/event objects
- one availability explain path
- one tombstone restore and reconcile proof
- one recurring write-scope proof
- one field-ownership conflict proof per adapter
- one identity-promotion proof

These proving flows are the evidence that the membrane, ownership, migration, and adapter boundaries are real.

# Proving Ladder

The proving flows map onto the Phase 57 test ladder.

Required carry-through terms:

- test ladder
- fake adapter
- real smoke

Expected minimums:

- fake adapter integration for ownership, dry-run, tombstone, and write-intent paths
- real smoke for minimal provider auth and bounded sync proof
- migration and replay tests for migration artifact import

# Availability, Tombstone, And Read-Only Coverage

Migration and proving must explicitly cover:

- Google Calendar availability inputs and explain behavior
- tombstone preservation and restore behavior
- read-only enforcement
- stale version handling
- migration artifact import for representative legacy state

These are required because they are the places most likely to produce silent corruption if left vague.

# Backend-Safe Carry-Through

Migration and compatibility rules must remain backend-safe:

- storage-agnostic
- serde-versioned
- compatible with optimistic concurrency
- explicit about stale version outcomes
- compatible with `CredentialProvider` and `SecretStore` boundaries

Migration logic must also remain testable through in-memory, fake adapter, and real smoke layers rather than only through one production-shaped environment.

# Non-Goals

This contract does not:

- define full implementation details for every adapter phase
- allow permanent compatibility DTO shadow systems
- replace milestone verification with prose confidence

# Summary

`0.5` migration and cutover law is explicit:

- migration artifact shape is versioned and machine-readable
- replay is inspectable and idempotent
- Compatibility DTO layer use is temporary and bounded by removal criteria
- proving flows must cover Todoist backlog, Google Calendar, availability, tombstone behavior, read-only enforcement, and migration artifact import across the test ladder, fake adapter, and real smoke tiers
