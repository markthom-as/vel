# Purpose

Define the `0.5` registry objects that govern shipped and installed system capabilities without collapsing them into ordinary content objects.

# Registry Objects

`Module`, `Skill`, and `Tool` are canonical registry objects.

They are durable enough for governance, audit, policy attachment, explainability, deprecation, and discovery, but they are not ordinary content objects like `Task` or `Event`.

Registry objects remain registry objects even when their definitions are manifest-backed.

# Registry Objects Versus Content Objects

The distinction is intentional:

- content objects are durable semantic world records such as `Task`, `Event`, and `Workflow`
- registry objects are durable system-capability records such as `Module`, `Skill`, and `Tool`

Registry objects must not be left as ghost assets on disk, and they must not be flattened into content-object semantics.

# Manifest-Backed Definitions And Optional Persisted Overlays

`0.5` treats registry definitions as manifest-backed with optional persisted overlays.

That means:

- a module manifest is an input artifact
- canonical registry identity is durable in the registry substrate
- persisted overlays may track enablement, policy posture, local metadata, version pinning, deprecation, or operator notes

Manifest-backed does not mean manifest-only.

Optional persisted overlays exist so registry objects can remain canonical enough for policy and explain without forcing every local adjustment back into static manifests.

# Canonical Registry IDs

Registry IDs are stable human-readable semantic IDs.

Examples:

- `module.integration.todoist`
- `module.integration.google-calendar`
- `skill.core.daily-brief`
- `tool.object.get`

Registry IDs are semantic referents for docs, policy, workflow declarations, audit output, and explain surfaces.

Required registry object metadata:

- `id`
- `namespace`
- `slug`
- `display_name`
- `version`
- `status`
- `capability_requests`
- `manifest_ref`

Do not rely on parsing the ID as the sole metadata source. The canonical ID is stable, but structured fields remain first-class.

# Module

`Module` is the governed bundle that contributes registry-visible capabilities and manifest-backed assets.

A module may declare:

- `Skill` records
- `Tool` records
- seeded `Workflow` definitions
- adapter registrations
- policy bundles
- config presets

Modules remain subordinate to core semantics, the action membrane, and policy evaluation.

# Skill

`Skill` is a canonical registry object representing a governed capability package.

Skills are invokable by workflows and user-facing entrypoints through the membrane. They do not become durable content objects just because they can be versioned or audited.

Skills do not call raw tools directly. Skill execution must remain mediated by grants, policy, and action contracts.

# Tool

`Tool` is a canonical registry object representing a governed primitive capability surface.

Tools are discovered, versioned, and audited through the registry. They do not bypass policy simply because they are low-level.

# Workflow Relationship

`Workflow` is not a registry object. It is a canonical content object.

Modules may seed workflows, reference workflows, or declare workflow manifests, but the stable durable workflow referent is a canonical `Workflow` content object stored in canonical storage.

This keeps modules, skills, tools, and workflows on one runtime/governance plane without collapsing their lifecycle classes.

# Backend-Safe Registry Roles

The registry contract must be implementable through explicit backend roles:

- `ManifestSource`
- `RegistryLoader`
- `RegistryReconciler`
- `RegistryStore`

Expected responsibilities:

- `ManifestSource`: load manifests from bundled assets, filesystem paths, or package sources
- `RegistryLoader`: parse and validate manifests into canonical registry entities
- `RegistryReconciler`: compare manifest-backed definitions against persisted registry state and produce idempotent updates
- `RegistryStore`: persist canonical registry identity and optional persisted overlays

These roles are required so the Rust backend can remain storage-agnostic, deterministic, and testable.

# Deterministic Bootstrap And Idempotent Registration

Bootstrap must be deterministic and idempotent.

Required rules:

- registry seeding precedes workflow seeding
- the same manifest input must produce the same canonical registry IDs
- rerunning bootstrap must not duplicate registry objects
- reconciliation state must remain inspectable rather than implicit

The registry layer must support deterministic bootstrap for bundled modules and idempotent re-registration for upgrades or local restarts.

# Feature Gating

Registry contracts must support feature gating explicitly.

Examples:

- provider modules gated by `todoist` or `google-calendar`
- storage implementations gated by `sqlite` or `postgres`
- runtime helpers gated by `background-sync`

Feature gating must not change the canonical meaning of registry objects. It only affects availability, activation, or compilation surface.

# Activation Posture

Registration is not execution.

Activation makes a registry object available for governed use. Invocation remains a separate policy-governed runtime event.

An installed or activated module does not gain direct permission to mutate canonical or provider state by default.

# Non-Goals

This contract does not:

- define workflow runtime execution
- define provider API payloads
- allow modules to bypass the membrane
- treat registry objects as ordinary user-authored content

# Summary

`0.5` has one canonical registry layer:

- `Module`
- `Skill`
- `Tool`

They are manifest-backed with optional persisted overlays, identified by stable human-readable registry IDs, and loaded through explicit backend roles such as `ManifestSource`, `RegistryLoader`, and `RegistryReconciler` under feature gating, deterministic bootstrap, and idempotent reconciliation rules.
