# Purpose

Define the durable semantic substrate for milestone `0.5` before storage, adapters, modules, and workflows spread implementation assumptions through the codebase.

# Object Classes

`0.5` uses four distinct entity classes:

- `content`: durable canonical objects that represent the semantic world operators and integrations work with
- `registry`: durable canonical registry objects that identify modules, skills, and tools
- `read_model`: rebuildable derived views optimized for explain/query/use, not canonical source of truth
- `runtime`: control-plane and execution records such as runs, audits, grants, approvals, sync jobs, and write intents

These classes must remain distinct. “Canonical” does not mean “one undifferentiated blob.”

# Canonical Content Objects

Canonical content objects in `0.5` are:

- `Task`
- `Project`
- `Event`
- `Calendar`
- `Thread`
- `Message`
- `Person`
- `Tag`
- `IntegrationAccount`
- `SyncLink`
- `Workflow`
- `Config`

These records live in the durable semantic world and are the system of record for backend behavior.

# Canonical Registry Objects

Canonical registry objects in `0.5` are:

- `Module`
- `Skill`
- `Tool`

Registry objects have durable canonical identity and governance metadata, but they are not ordinary user-authored content objects. Their definitions may be manifest-backed while their identity, policy posture, version pinning, enablement, and deprecation state remain canonical.

# Read Models

Read models are derived, rebuildable, and non-authoritative.

Examples:

- `Availability`
- projection caches
- compact summary views

Read models may be cached or materialized, but they do not redefine canonical truth.

# Runtime And Control Records

Runtime/control records are not part of the canonical content substrate.

Examples:

- `RunRecord`
- `AuditEntry`
- `Grant`
- `Approval`
- `SyncJob`
- `WriteIntent`
- projection/materialization records

These records are durable adjacent infrastructure, not canonical semantic objects.

# Universal Durable Envelope

All durable canonical objects share one narrow envelope. The envelope is storage-agnostic, serde-compatible, versioned, and suitable for typed Rust newtype-backed IDs.

Required envelope fields:

- `id`
- `object_type`
- `object_class`
- `schema_version`
- `created_at`
- `updated_at`
- `status`
- `provenance`
- `facets`

Optional shared envelope fields:

- `deleted_at`
- `archived_at`
- `source_summary`

The envelope must stay narrow. Object-specific domain data belongs in typed object payloads, not in an ever-growing universal wrapper.

Example content IDs:

- `task_01hxyz...`
- `event_01hxyz...`
- `thread_01hxyz...`
- `workflow_01hxyz...`

Typed prefixed IDs are required for content objects. The Rust backend should represent them as typed newtype wrappers even when wire forms are strings.

# ID Policy

Content objects use typed prefixed IDs.

Examples:

- `task_...`
- `project_...`
- `event_...`
- `calendar_...`
- `thread_...`

Registry objects use stable human-readable semantic IDs.

Examples:

- `module.integration.todoist`
- `module.integration.google-calendar`
- `skill.core.daily-brief`
- `tool.object.get`

This split is intentional:

- content IDs optimize for lifecycle safety and cardinality
- registry IDs optimize for stable explanation, policy, invocation, and documentation

# source_summary

`source_summary` is an optional shared envelope field.

It exists only as a derived convenience summary for explain/debug/query ergonomics. It may include:

- linked provider names
- active link count
- primary upstream source indicator
- last sync timestamp
- sync health summary
- tombstone or conflict flags

`source_summary` is not linkage truth and must never replace `SyncLink` plus typed relations.

# Canonical Boundaries

The following stay explicitly outside the canonical content substrate:

- credentials
- raw secrets
- grants
- approvals
- audit entries
- sync jobs
- write intents
- execution run state

Credentials are not canonical objects. `IntegrationAccount` is canonical; credentials live behind secure external secret state and explicit backend traits.

# Contract Posture For Rust

The `0.5` object model is backend-safe by design:

- storage-agnostic domain contracts
- serde-backed versioned wire forms
- typed Rust ID newtype posture
- feature-gated provider and storage implementations
- no platform-specific handles in core durable contracts

Downstream implementations must preserve this posture rather than allowing SQL shape, provider payloads, or platform APIs to leak upward into the domain.

# Linkage Boundary

Canonical linkage truth is external through `SyncLink` objects and typed relations. See [0.5-canonical-relations-and-linkage.md](/home/jove/code/vel/docs/cognitive-agent-architecture/architecture/0.5-canonical-relations-and-linkage.md).

The machine-readable draft for the shared envelope lives at [0.5-canonical-object-envelope.schema.json](/home/jove/code/vel/config/schemas/0.5-canonical-object-envelope.schema.json).

# Non-Goals

This contract does not:

- define provider adapter mappings in detail
- define runtime execution semantics
- flatten registry, read_model, and runtime into one object family
- allow arbitrary ungoverned extension blobs

# Summary

`0.5` has one durable canonical object model:

- `content`
- `registry`
- `read_model`
- `runtime`

with one narrow envelope, typed prefixed IDs for content objects, stable registry IDs for registry objects, and an optional `source_summary` that stays derived rather than redefining linkage truth.
