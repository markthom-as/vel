# Purpose

Define the narrow provider boundary for the two proving adapters in `0.5` so later implementation cannot widen into generic connector sprawl or let provider semantics replace the canonical core.

# Adapter Law

Todoist and Google Calendar are sources and sync partners.

They are not ontology providers.

Vel owns:

- canonical objects
- cross-object relations
- durable IDs
- `source_summary`
- task semantics and other `Vel-only` enrichment
- audit, run, and `WriteIntent` records

Providers own their provider-specific data, remote identities, remote-side lifecycle state, and raw payload semantics.

# Ownership Vocabulary

Provider boundaries use one explicit field-ownership vocabulary:

- `source-owned`
- `shared`
- `Vel-only`

These terms must line up with the checked-in ownership matrix and runtime conflict evaluation.

# Todoist Boundary

Todoist is the proving adapter for canonical `Task`, `Project`, `Tag`, `AttachedCommentRecord`, and task history ingestion.

## Todoist Task Ownership

Typical Todoist task field posture:

- `source-owned`: provider remote ID, Todoist section metadata, Todoist ordering hints, upstream completion timestamps
- `shared`: title, description, canonical priority, due, parent relation, primary project/container relation, tags
- `Vel-only`: `task_semantics`, derived analytics, workflow-local annotations, cross-object links, local explanation metadata

Todoist sections remain provider-facet metadata in `0.5`, not canonical top-level objects.

Todoist labels map into canonical tags, while exact Todoist label identity remains preserved in provider facets.

Todoist backlog import is intentionally broad. `0.5` uses Todoist backlog as the proving path for full-history-ish task state, comments, and eventful churn tracking rather than a narrow rolling window.

## AttachedCommentRecord

Todoist task comments map to `AttachedCommentRecord`.

`AttachedCommentRecord` is:

- attached, not standalone
- durable enough for canonical inspection
- shared across providers
- extensible through provider facets

It does not become a free-floating message/thread ontology in this milestone.

# Google Calendar Boundary

Google Calendar is the proving adapter for canonical `Calendar`, `Event`, attendance, recurrence, and availability inputs.

Google maps into the native calendar core. It does not redefine `Event`, `Calendar`, or availability truth.

## Google Event Ownership

Typical Google event field posture:

- `source-owned`: remote calendar ID, remote event ID, upstream recurrence metadata, provider transparency payload, organizer/source identity payload
- `shared`: title, description, start/end timing, attendee participation, recurrence overrides, explicit location payload, cancellation/tombstone state
- `Vel-only`: cross-object relations, explain metadata, derived availability projections, local semantic enrichment

## Person And Participation

Google attendees use canonical `Person` links where identity is resolvable.

Participation is canonical and includes:

- `Person` or provider stub reference
- participation response status
- organizer flag
- self flag
- optional flag
- resource flag

If no durable person identity exists, the attendee may remain a provider stub. Promotion into canonical `Person` must preserve provenance and avoid destructive replacement.

# Recurrence And Edit Scope

Recurring events use one explicit posture:

- recurring series canonical record
- explicit override records only when observed or changed
- queryable canceled overrides

Recurring edit scope in `0.5` is explicit:

- `this occurrence`
- `entire series`

`this occurrence` and `entire series` must remain distinct in explain, policy, and write handling.

`this and following` is not part of the `0.5` write contract.

# Sync Windows And Import Posture

Default Google Calendar sync posture is bounded:

- `past 90 days / future 365 days`

This bounded window is the default proving posture for import, explain, and smoke tests.

Todoist does not use the same bounded-window rule. Todoist backlog posture remains broad enough to preserve task history, comments, churn analysis, and full task state continuity.

# Tombstone Rules

Tombstone behavior is explicit and shared across both proving adapters.

Rules:

- upstream deletes become canonical tombstones
- tombstones stay hidden from default queries
- tombstones remain available for explain, audit, and migration artifact import
- restored upstream objects prefer revival of existing canonical identity when linkage confidence is strong

# Policy Presets

The provider-boundary vocabulary uses these preset names:

- `read_only`
- `conservative_sync`
- `manual_confirm`
- `local_enrichment_only`

Expected posture:

- `read_only`: import and inspect only
- `conservative_sync`: limited shared-field sync with strict ownership and confirmation posture
- `manual_confirm`: outward mutations require explicit human confirmation
- `local_enrichment_only`: provider import allowed, Vel-only enrichment stays local and never round-trips automatically

# Read-Only And Concurrency Posture

read-only enforcement is mandatory at workspace, module, and integration-account boundaries.

Concurrency-sensitive operations must assume:

- optimistic concurrency
- stale version detection
- explicit reconcile requirement on ownership conflict

No adapter may silently overwrite stale remote state or canonical state just because the provider allows a write API call.

# Secret And Runtime Boundaries

Canonical `IntegrationAccount` records do not store raw credentials.

Provider adapters must use:

- `CredentialProvider`
- `SecretStore`

The core domain may observe capability posture, auth state, and account identity, but it must not depend on raw token handling.

Provider adapters remain behind runtime traits and execution seams rather than leaking provider SDK shape into the domain layer.

# Proving Flows

These provider boundaries are only considered real if they support milestone proving flows such as:

- Todoist backlog import
- Google Calendar bounded import
- availability explain
- tombstone restore and reconcile
- recurring write scope proof for `this occurrence` and `entire series`
- read-only enforcement
- fake adapter contract tests
- real smoke paths for minimal auth and sync

# Non-Goals

This contract does not:

- define Gmail, Docs, Drive, or Contacts behavior
- define UI inspection surfaces in detail
- expand Google into provider-first calendar ontology
- elevate sections/lists into canonical top-level objects in `0.5`

# Summary

`0.5` has two narrow proving adapters:

- Todoist for tasks, comments, backlog, and churn history
- Google Calendar for calendars, events, attendance, recurrence, and availability inputs

Both remain subordinate to the canonical core, use explicit `source-owned` / `shared` / `Vel-only` ownership, obey read-only enforcement, respect optimistic concurrency and stale version handling, and depend on `CredentialProvider` and `SecretStore` rather than raw secret sprawl.
