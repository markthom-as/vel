# Vel — Current Status

This document is the **canonical repo-wide implementation ledger** for Vel.

- All repo-wide implementation status questions resolve here.
- Subsystem docs may contain additional detail, but must not contradict this file.
- Product vision docs, MVP docs, and ticket packs are not implementation truth.
- If another doc conflicts with this file about what is shipped now, follow this file.

This file should answer four questions quickly:

- what is implemented now
- what is partial
- what is planned next
- what is intentionally deferred

## Implemented

- **Capture storage** — Insert captures, list by ID, lexical search (FTS5).
- **Artifacts API** — Create and fetch artifacts; storage layer with metadata.
- **Run/event schema and inspection** — `runs`, `run_events`, `events` tables; `vel runs` (with `--kind`, `--today`, `--limit`), `vel run inspect <id>`, `vel run status <id> <status>` (e.g. retry_scheduled, blocked, plus `--retry-after-seconds`, `--retry-at`, `--reason`, `--blocked-reason`, `--allow-unsupported-retry`); `GET /v1/runs?limit=&kind=&today=`, `GET /v1/runs/:id`, `PATCH /v1/runs/:id` (body supports `status`, `retry_at`, `retry_after_seconds`, `reason`, `blocked_reason`, `allow_unsupported_retry`). Run list/detail now also expose per-kind automatic retry policy metadata so operators can see whether worker retry is supported.
- **Run retry lifecycle** — `retry_scheduled` is now actionable for supported run-backed flows. Worker automation picks up due retries, resets the run safely, appends `run_retry_scheduled` / `run_requeued`, and re-executes existing runs for context generation and synthesis using the original run input. Unsupported automatic retries now transition to `blocked` with a persisted `blocked_reason`.
- **Doctor diagnostics** — `vel doctor` and `GET /v1/doctor` with structured checks (DiagnosticCheck + DiagnosticStatus).
- **Context endpoints (run-backed)** — `GET /v1/context/today`, `GET /v1/context/morning`, `GET /v1/context/end-of-day` create a run, compute from orientation snapshot, write a managed **context_brief** artifact (write temp → flush → fsync → rename; checksum, size_bytes, metadata_json with `generator`, `context_kind`, `snapshot_window`), link run → artifact and artifact → capture refs, append run events (`run_created`, `run_started`, `context_generated`, `artifact_written`, `refs_created`, `run_succeeded`), then transition to succeeded/failed.
- **Run timing** — API and CLI expose `duration_ms` (derived from started_at/finished_at) on run list and run detail.
- **Runtime invariants** — Documented in runtime-concepts: run_started precedes run_succeeded; artifact_written/refs_created before run_succeeded; run_failed does not produce artifact refs; run_succeeded implies artifact durability.
- **CLI** — `vel health`, `vel capture`, `vel journal mood`, `vel journal pain`, `vel search`, `vel today`, `vel morning`, `vel end-of-day`, `vel doctor`, `vel inspect capture <id>`, `vel runs`, `vel run inspect <id>` (duration, artifact size e.g. 3.4KB, event timestamps in CLI).
- **Container packaging** — Dockerfile plus `docker-compose.yml` for Docker or Podman NAS/local daemon deployment of `veld` with persistent `/data` storage, port `4130`, and CLI-backed healthcheck.
- **Nix-hosted runtime path** — `shell.nix` now includes the Rust/Node/OpenSSL toolchain needed to build and run Vel reproducibly with `nix-shell`, plus repo-level `make nix-*` helper targets.
- **Crate boundaries** — Domain types (ContextCapture, SearchResult, OrientationSnapshot) live in `vel-core`; storage returns them; API layer maps to DTOs. `vel-storage` does not depend on `vel-api-types`.
- **Run events uniqueness** — `(run_id, seq)` unique on `run_events`.
- **Typed run payloads** — Run and RunEvent use `serde_json::Value` in domain/API; DB remains TEXT; (de)serialization at storage boundary.
- **Run transitions** — Immutable transitions in `vel-core` (start/succeed/fail/cancel return new `Self`).
- **Service layer** — Context generation and doctor logic live in `veld` services; routes are thin.
- **Artifact storage kind** — `storage_kind` (managed | external) in schema, core, and API.
- **Global events (optional)** — `DAEMON_STARTED` and `CONFIG_LOADED` emitted on veld startup; `SCHEMA_MIGRATION_COMPLETE` emitted after migrations run (with `schema_version` in payload). Role remains system audit log.
- **Run statuses** — `RunStatus` includes `RetryScheduled` and `Blocked`, with live API/CLI/operator support for manual updates and worker-driven retry pickup for supported run-backed flows.
- **Service-level result type** — Context generation returns `ContextRunOutput<T>` (run_id, artifact_id, context_kind, data); routes map `.data` to API response.
- **Canonical doc names** — Renamed `docs/vel-*.md` to `docs/*.md` (e.g. `runtime-concepts.md`, `data-model.md`); README and AGENTS.md updated.
- **Vocabulary / glossary** — `docs/vocabulary.md` is now the first-class terminology appendix and index for Vel. The canonical programmatic vocabulary lives in `vel-core::vocabulary`, and `vel-cli` command-language registry/completion consume that shared term set rather than duplicating DSL/domain aliases.
- **Commitments (Phase A)** — Schema (`commitments` table + indexes), domain type in `vel-core` (`Commitment`, `CommitmentId`, `CommitmentStatus`), storage CRUD, API (POST/GET /v1/commitments, GET/PATCH /v1/commitments/:id), CLI (`vel commitments`, `vel commitment add/done/cancel/inspect`). Capture promotion: captures with `capture_type == "todo"` auto-create an open commitment (source_type capture, commitment_kind todo). **Commitment dependency graph:** storage `insert_commitment_dependency`, `list_commitment_dependencies_by_parent/child`; API GET/POST /v1/commitments/:id/dependencies; CLI `vel commitment dependencies <id>`, `vel commitment add-dependency <parent> <child> [--type blocks]`. Spec: [docs/specs/commitments.md](specs/commitments.md).
- **Phase B — Signal ingestion** — `signals` table; storage `insert_signal`, `list_signals`; `signals.source_ref` is now used for replay-safe adapter dedupe when a stable external identity exists. API POST/GET /v1/signals; adapters: calendar (ICS path/URL → calendar_event signals), Todoist (snapshot JSON → commitment reconciliation plus lifecycle-aware `external_task` signals; completed tasks resolve linked commitments and unchecked tasks reopen/update them), activity (snapshot JSON for `shell_login` / `computer_activity` / `idle_state`, with `vel_invocation` fallback when no snapshot path is configured, and replay-safe `source_ref` for snapshot-backed events), health (snapshot JSON → replay-safe `health_metric` signals with metric type/value/unit/source metadata), git (snapshot JSON → replay-safe `git_activity` signals), notes (markdown/plaintext file or directory sync → replay-safe `note_document` captures and signals), transcripts (snapshot JSON → `assistant_transcripts` rows + `assistant_message` signals with replay-safe dedupe), messaging (snapshot JSON → replay-safe `message_thread` signals carrying waiting-state, urgency, and scheduling metadata); CLI `vel sync calendar/todoist/activity/health/git/notes/transcripts/messaging`; config `calendar_ics_url`, `calendar_ics_path`, `todoist_snapshot_path`, `activity_snapshot_path`, `health_snapshot_path`, `git_snapshot_path`, `notes_path`, `transcript_snapshot_path`, `messaging_snapshot_path`. On macOS, `veld` now auto-discovers local `activity` / `health` / `git` / `messaging` / `notes` / `transcripts` sources under `~/Library/Application Support/Vel/...` when present and performs a startup bootstrap sync so those sources reach current context without manual path setup.
- **Pain and mood journaling** — `POST /v1/journal/mood` and `POST /v1/journal/pain` create private `mood_log` / `pain_log` captures and matching structured signals; CLI `vel journal mood <score>` and `vel journal pain <severity>` provide the same path with optional labels/locations/notes/source tags. This is a daily-orientation feature, not a medical analytics subsystem. Spec: [docs/specs/vel-pain-and-mood-journaling-spec.md](specs/vel-pain-and-mood-journaling-spec.md).
- **Phase C — Inference** — `inferred_state` table; service reads signals + commitments, computes morning state (inactive/awake_unstarted/underway/engaged/at_risk), meds_pending, prep_window_active; persists one inferred_state record; emits STATE_CHANGED.
- **Phase D — Nudges** — `nudges` table; nudge engine creates meds_not_logged, meeting_prep_window, morning_drift, and response_debt (gentle/warning level); API GET /v1/nudges, GET /v1/nudges/:id, POST /v1/nudges/:id/done, POST /v1/nudges/:id/snooze; CLI `vel nudges`, `vel nudge inspect <id>`, `vel nudge done <id>`, `vel nudge snooze <id> --minutes 10`.
- **Phase E — Notification** — Nudges surfaced via CLI (`vel nudges`) and API; desktop/watch stubbed for later.
- **Phase F — Weekly synthesis** — POST /v1/synthesis/week: run-backed (RunKind::Synthesis), writes **weekly_synthesis** artifact with canonical shape (week_start, week_end, summary, completed/open_commitment_ids, top_threads, drift_patterns, vel_self_review, suggested_adjustments); CLI `vel synthesize week`. Spec: [vel-weekly-synthesis-spec.md](specs/vel-weekly-synthesis-spec.md).
- **Phase G — Observability** — Events emitted: SIGNAL_INGESTED, STATE_CHANGED, NUDGE_GENERATED, NUDGE_RESOLVED; evaluate endpoint POST /v1/evaluate runs inference + nudge engine; CLI `vel evaluate`.
- **Schema (migrations 0012–0022)** — `current_context` (singleton), `context_timeline`, `commitment_dependencies`, `commitment_risk`, `nudge_events`, `suggestions`, `threads`, `thread_links`, `vel_self_metrics`, `assistant_transcripts`; `signals.source_ref` added. See [docs/specs/vel-migrations-and-schema-spec.md](specs/vel-migrations-and-schema-spec.md).
- **Persistent current context** — Inference engine writes canonical context (mode, morning_state, inferred_activity, next_commitment_id, prep/commute windows, meds_status, active_nudge_ids, top_risk_commitment_ids, global_risk_level/score, leave_by_ts, next_event_start_ts, health_summary, git_activity_summary, mood_summary, pain_summary, note_document_summary, assistant_message_summary, messaging-awareness summary fields, attention/drift fields) to `current_context`; material-change detection appends to `context_timeline`; GET /v1/context/current, GET /v1/context/timeline; GET /v1/now and GET /v1/explain/context now also expose typed mood/pain source summaries when present; CLI `vel context` (show), `vel context timeline`; explain routes remain read-only and now expose both `signals_used` ids, hydrated signal summaries, and typed source summaries. The web Context panel now consumes explain-oriented context/drift data instead of only dumping the raw persisted context JSON, and also surfaces the current source summaries directly. Spec: [vel-current-context-spec.md](specs/vel-current-context-spec.md).
- **Nudge events** — Append-only `nudge_events` on nudge create, done, snooze, reactivation, and escalation; supports explainability and durable intervention lifecycle semantics.
- **Explainability** — GET /v1/explain/nudge/:id (nudge + inference/signals snapshot + lifecycle history); GET /v1/explain/context (including typed source summaries for git/note/transcript inputs); GET /v1/explain/commitment/:id (commitment + risk snapshot + why in context); GET /v1/explain/drift (attention_state, drift_type, reasons, signals_used, commitments_used); CLI `vel explain nudge <id>`, `vel explain context`, `vel explain commitment <id>`, `vel explain drift` (all support `--json`).
- **Policy engine** — Resolution policies run first (auto-resolve meds/prep/drift/response-debt nudges when conditions no longer hold); each nudge creation stores structured explanation in `metadata_json` (policy, decision, level, reasons). Spec: [vel-policy-engine-spec.md](specs/vel-policy-engine-spec.md).
- **Signal adapters** — On capture create, a `capture_created` signal (source `vel`) is inserted in addition to CAPTURE_CREATED event; POST /v1/signals accepts `vel_feedback` for user feedback. Spec: [vel-signal-adapter-spec.md](specs/vel-signal-adapter-spec.md).
- **Policy config** — Loaded from `config/policies.yaml` at startup (or `VEL_POLICIES_PATH`); missing/malformed file fails startup. Injected into nudge engine (default_prep_minutes, commute_leave_time thresholds). Spec: [vel-agent-next-steps-policy-config-commute.md](specs/vel-agent-next-steps-policy-config-commute.md).
- **commute_leave_time nudge** — Fires only when calendar event has `travel_minutes`; leave_by = event_start - travel_minutes; escalation ladder (gentle/warning/danger) from config; resolution on event start or commitment resolve.
- **Context explain** — GET /v1/explain/context returns `signals_used`, `commitments_used`, `risk_used`, `reasons`; inference stores used ids in current context JSON.
- **Thread graph** — Storage: insert_thread, get_thread_by_id, list_threads, update_thread_status, insert_thread_link, list_thread_links. API: GET/POST /v1/threads, GET/PATCH /v1/threads/:id, POST /v1/threads/:id/links. CLI: `vel threads`, `vel thread inspect/close/reopen`. Spec: [vel-thread-graph-spec.md](specs/vel-thread-graph-spec.md).
- **Risk engine compute** — Consequence, proximity, dependency pressure (no uncertainty/progress-penalty in first version). POST /v1/evaluate runs inference → risk::run → nudge engine and populates `commitment_risk`; GET /v1/risk returns current risk state, GET /v1/risk/:id returns risk for a single commitment; CLI `vel risk`, `vel risk <commitment_id>`. Spec: [vel-risk-engine-spec.md](specs/vel-risk-engine-spec.md), [vel-agent-next-implementation-steps.md](specs/vel-agent-next-implementation-steps.md).
- **Attention/drift in current context** — Inference writes `attention_state`, `drift_type`, `drift_severity`, `attention_confidence`, `attention_reasons` into context JSON (morning_drift, prep_drift heuristics). Spec: [vel-attention-and-drift-detection-spec.md](specs/vel-attention-and-drift-detection-spec.md).
- **Suggestions (steering loop)** — Storage: insert/list/get/update suggestions plus evidence rows, dedupe-key lookups, and richer metadata (`title`, `summary`, `priority`, `confidence`, `decision_context_json`, `evidence_count`). API: GET /v1/suggestions, GET /v1/suggestions/:id, GET /v1/suggestions/:id/evidence, PATCH /v1/suggestions/:id, POST /v1/suggestions/:id/accept, POST /v1/suggestions/:id/reject; inspect returns full decision context and the evidence list is also queryable directly. CLI: `vel suggestions`, `vel suggestion inspect/accept/reject/modify`. Two types (increase_commute_buffer, increase_prep_window) are triggered by repeated resolved nudge evidence, persist evidence-backed steering metadata, suppress duplicate pending suggestions by dedupe key, and on acceptance write adaptive policy overrides that the next evaluation consumes for prep/commute timing. Accepted suggestions now expose their latest feedback outcome directly (`accepted_and_policy_changed` vs `accepted_no_effect`) so operators can distinguish real runtime changes from acknowledged-but-non-binding proposals. Rejections can now persist an explicit operator reason in suggestion payload metadata, and repeated `rejected_not_useful` feedback suppresses that suggestion family more aggressively than a one-off incorrectness penalty. Active adaptive overrides are exposed through `/api/settings` and surfaced in the Settings UI. Spec: [vel-agent-next-implementation-steps.md](specs/vel-agent-next-implementation-steps.md).
- **Project synthesis** — POST /v1/synthesis/project/:slug; run_project_synthesis filters commitments by project, writes **project_synthesis** artifact (open_commitments, active_threads, completed_commitment_ids, etc.); CLI `vel synthesize project <name>`. Spec: [vel-agent-next-implementation-steps.md](specs/vel-agent-next-implementation-steps.md).
- **Integration tests** — Commute policy: no commute nudge when calendar event has no travel_minutes; commute nudge fires with travel_minutes in leave window; context explain returns signals_used, commitments_used, reasons; resolution order: resolved nudge stays resolved after second evaluate. **Canonical day fixture:** shared fixture (Meeting with Dimitri, prep 30, travel 40, meds/prep/commute commitments); §6.1 context assertions (prep/commute window, meds_status, mode); §6.2 risk assertions (non-empty risk list); §6.3 nudge snooze suppresses until snoozed_until, event start suppresses commute nudge; §6.5 explain references commitments and signals; §6.6 project synthesis artifact; Variant A (success: meds done removes meds nudge); Variant B (drift: commute danger + drift/attention in context); Variant C (suggestion: repeated commute danger triggers increase_commute_buffer).
- **Feedback compliance (review round)** — Evaluate order: risk → inference → nudges so context incorporates risk snapshot; inference reads `commitment_risk` for `top_risk_commitment_ids`, `risk_used`, `global_risk_level`/`score`; risk factors include `dependency_ids`; policy engine uses risk for meeting_prep_window level (warning when meeting_risk >= high); suggestion triggers on repeated evidence (commute danger, prep resolved) via `suggestions::evaluate_after_nudges` after evaluate; observability: `vel explain context`, `vel explain commitment <id>`, `vel explain drift`. Events table present (migration 0004). No new subsystems added.
- **Latest repo feedback** — [vel-latest-repo-feedback.md](reviews/vel-latest-repo-feedback.md). **Explain routes read-only:** `explain_commitment` no longer calls `risk::run()`; it reads latest persisted risk via `list_commitment_risk_recent(id, 1)`. Repo-level read-boundary coverage now also asserts that `GET /v1/now` and `GET /v1/suggestions*` do not create runs or mutate persisted state. **First event = next future event:** inference uses the nearest upcoming event at or after `now`, with fallback to most recent past event for “currently active” display. **Commute nudge resolution:** resolved only when event has started or the related commitment is resolved (no global `open_commitments.is_empty()`).
- **Repo-feedback follow-through** — Read routes are now covered by a repo-level non-mutation test across explain/context endpoints. Inference is being refactored into explicit collect/derive/persist stages with deterministic selectors instead of insertion-order fallbacks. Risk now has a typed core contract (`RiskFactors`, `RiskSnapshot`) and missing-risk context is explicit (`global_risk_level = "unknown"`, `global_risk_score = null`, `global_risk_missing = true`) rather than synthesized. Nudge lifecycle now supports expired-snooze reactivation and in-place escalation for higher-urgency active nudges. Websocket envelopes now use RFC3339 `timestamp`, while resource DTO timestamp fields remain Unix seconds.
- **Architecture boundaries (detailed next steps)** — Nudge engine refactored to consume **current context + risk + nudges + policy** only: no recomputation of meds/prep/morning/messaging state from raw signals. It reads `get_current_context()`, `list_commitment_risk_latest_all()`, `list_nudges()`, policy config; uses context fields `meds_status`, `prep_window_active`, `commute_window_active`, `morning_state`, `leave_by_ts`, `next_event_start_ts`, `message_waiting_on_me_count`, `message_scheduling_thread_count`, and `message_urgent_thread_count` for resolution and creation. Inference writes those fields into context so commute and response-debt nudges can be computed from context. Single source of truth for present state is context reducer (inference).
- **Runtime loop control plane** — Runtime loops now have durable storage-backed state (`runtime_loops`), claim/complete semantics in storage/worker, and operator surfaces for inspection/control. Scheduled loops cover evaluate plus adapter sync for calendar, Todoist, activity, messaging, git, notes, and transcripts. API: `GET /v1/loops`, `GET /v1/loops/:kind`, `PATCH /v1/loops/:kind`. CLI: `vel loops`, `vel loop inspect <kind>`, `vel loop enable <kind>`, `vel loop disable <kind>`. The web Settings surface now exposes a Runtime Loops tab for enable/disable and interval updates. Route-level validation rejects invalid interval updates as `400` instead of surfacing storage validation as `500`.
- **Agent spec configuration** — `vel-config` now supports `agent_spec_path` in `vel.toml` / `VEL_AGENT_SPEC_PATH`, validates YAML agent-spec files, and exposes a sample spec pack at `config/agent-specs.yaml`. CLI `vel config show` now prints both LLM model paths and the effective agent-spec path so local operator/runtime setup is inspectable from one surface.
- **Cluster bootstrap discovery** — `GET /v1/cluster/bootstrap` now exposes a minimal bootstrap contract for planned multi-node Vel setups: node identity, current authority id/epoch, preferred sync base URL, and available Tailscale/LAN/localhost endpoints. `vel-config` now supports `node_id`, `node_display_name`, `tailscale_base_url`, and `lan_base_url`, and `vel config show` prints those effective values for local verification.
- **Client sync bootstrap and action lane** — `GET /v1/sync/bootstrap` now exposes a unified cache-hydration payload for current context, nudges, commitments, and cluster bootstrap metadata. `POST /v1/sync/actions` accepts batched low-risk client actions for `nudge_done`, `nudge_snooze`, `commitment_done`, `commitment_create`, and `capture_create`, and also carries structured `branch_sync_request` / `validation_request` intents into canonical signal storage for later execution.
- **Cluster inspection surfaces** — `GET /v1/sync/cluster` now exposes a minimal cluster-view payload with authority metadata, cluster view version, node state, and worker-capacity summaries backed by a durable `cluster_workers` registry. `GET /v1/cluster/workers` exposes the current cluster-visible worker presence/capacity view, and `POST /v1/sync/heartbeat` updates that registry with bounded worker-heartbeat metadata. The local worker row is now refreshed from live runtime state before cluster reads and work transitions, so published `current_load` and `queue_depth` reflect the running worker loop instead of a static placeholder. CLI now supports `vel sync bootstrap`, `vel sync cluster`, and `vel sync workers` for operator inspection.
- **Queued work receipts** — Routed branch-sync and validation requests now have a durable execution-receipt lane. `POST /v1/sync/work-assignments` claims a queued work request, `PATCH /v1/sync/work-assignments` updates receipt state (`started`, `completed`, `failed`, `cancelled`), `GET /v1/sync/work-assignments` lists receipts by `work_request_id` and/or `worker_id`, `GET /v1/sync/work-queue` lists pending queued items for a node/worker class with `attempt_count`, `claimable_now`, `claim_reason`, and `next_retry_at`, and `POST /v1/sync/work-queue/claim-next` claims the next eligible unit for a worker. Storage persists these receipts in `work_assignment_receipts`, duplicate queue requests reuse active or stale-reclaimable receipts instead of creating fresh routing signals, stale in-flight receipts can be reclaimed, and failed receipts now respect per-work-type retry policy from `config/policies.yaml` (`queued_work.validation`, `queued_work.branch_sync`) before becoming claimable again. Receipt state changes also emit durable signals (`work_assignment_claimed`, `work_assignment_started`, `work_assignment_completed`, `work_assignment_failed`, `work_assignment_cancelled`) for replay and inspection.
- **Background scheduler loop** — A runtime loop polls `POST /v1/sync/work-queue/claim-next`, auto-claims eligible units, and advances receipts so workers self-police retry/backoff without manual CLI hits; the loop publishes events via `GET /v1/loops` / `vel loops` for operator visibility and uses `POST /v1/sync/work-assignments`/`PATCH /v1/sync/work-assignments` for each claimed work unit. It now executes queued branch-sync units (in addition to validation work) when the node advertises the relevant capability, but execution stays bounded by the authority/receipt policy and per-work-type retry configuration so the loop preserves a single source of truth.
- **Capability-aware client sync requests** — Cluster/bootstrap payloads now also advertise node capabilities, repo branch-sync support, and environment-scoped validation profiles (`api`, `web`, `apple`, `repo`, `runtime`). The existing client sync action lane can now carry structured `branch_sync_request` and `validation_request` payloads, which are durably recorded as canonical signals for later execution/integration instead of being left as free-form client intent.
- **Chat interface** — Ticket pack 001–035 is implemented. Current chat runtime includes `/api/` conversations/messages/inbox/interventions/settings/integrations routes, `/ws`, frontend thread/inbox/settings surfaces, a first-class web `Now` view, a web suggestions surface, explicit chat/websocket transport DTOs in `vel-api-types`, a typed web decoder layer, a shared client query/cache layer for the main operator surfaces, and targeted backend/frontend tests. Websocket contract is `messages:new`, `interventions:new`, `interventions:updated`, `context:updated`, `runs:updated`, and `components:updated`. The web shell now exposes top-level `Now`, `Inbox`, `Threads`, `Suggestions`, `Projects`, `Stats`, and `Settings` surfaces, keeps `Settings` as a first-class page instead of a modal branch, adds a dedicated `Stats` observability page backed by existing runs/loops/components/integrations/context contracts, and gives the explain rail explicit `State` / `Why` / `Debug` modes. `Projects` is now a truthful placeholder surface in the shell while the backend project registry/workspace contract remains unimplemented. The web thread now uses optimistic user-message insertion with deterministic convergence against both websocket echoes and `POST /api/conversations/:id/messages` responses, and intervention snooze/resolve/dismiss actions now hide optimistically with rollback on failed POST. The Settings page now includes operator controls for Google Calendar and Todoist credentials/sync, local adapter status/sync cards for activity/git/messaging/notes/transcripts, editable local source-path controls that persist runtime overrides without a daemon restart, per-integration sync history, typed recovery guidance when an integration is missing credentials, disconnected, stale, or failing, and guidance-level action buttons that can trigger the relevant sync/connect/history/path-save flow directly from the warning state. The general settings surface now persists an IANA timezone, inference uses that timezone for local-day boundaries, and `GET /v1/now` returns the resolved timezone so operator timestamp formatting matches the backend’s day semantics. The `Now` surface loads from that dedicated read-only contract instead of stitching together four endpoints client-side, summarizes current context, schedule, tasks, attention, freshness, and recent source activity in one operator dashboard, refreshes immediately when evaluate/sync flows broadcast `context:updated`, refetches on browser focus plus a background interval, prioritizes Todoist/open tasks by urgency instead of insertion order, shows explicit degraded-source warnings while keeping stale data visible, carries integration recovery guidance through freshness banners/source rows, adds direct `Re-run evaluate` / `Sync` actions for retryable degraded sources, deep-links non-retryable integration warnings into the matching Settings card, distinguishes “no calendars selected” from “no upcoming events,” renders prep/travel/leave-by details directly in the schedule, and exposes a debug drawer for the raw snapshot fields. The suggestions surface now lets operators inspect evidence-backed steering proposals and accept/reject them without leaving the web shell. The Context panel is explain-backed, now surfaces typed source summaries for git/note/transcript inputs, and message provenance includes real linked objects plus card/intervention-derived evidence instead of placeholder-only arrays. Follow-up work remains broader provenance/policy hydration when stronger message-to-signal linkage exists, plus purpose-built Stats and Projects backend contracts. ## Partial

- **Global events** — Startup events (`DAEMON_STARTED`, `CONFIG_LOADED`, `SCHEMA_MIGRATION_COMPLETE`) implemented; runtime `config_updated` on hot reload not implemented (role remains narrow and documented).
- **Uncertainty runtime breadth** — Durable storage, suggestion deferral, and operator inspection are implemented; inference/chat uncertainty recording and calibration outcomes are not yet implemented.
- **Apple bootstrap clients** — `clients/apple` now has bootstrap iPhone/watchOS/macOS shells with shared VelAPI transport, endpoint candidate resolution (`vel_tailscale_url`, `vel_lan_base_url`, `vel_base_url`, localhost fallback), cluster-bootstrap-aware connection display, cached context/nudges/commitments, and queued offline actions for nudge/commitment/capture flows. Bootstrap metadata now also exposes branch-sync capability and validation profiles so richer clients can tell whether a node can handle repo sync or environment-specific checks. `VelMac` also exports local activity/health/messages snapshots into `~/Library/Application Support/Vel/...` and can trigger daemon sync for those local sources. This is still bootstrap-level client work, not a full offline-capable Apple product.
- **Cluster/sync runtime** — Distributed/swarm work is now partial rather than purely planned. Shipped pieces are bootstrap/cluster inspection surfaces, unified client sync bootstrap/action batching, Tailscale-aware endpoint selection, a durable heartbeat-backed worker registry with expiry, first-pass placement for queued validation/branch-sync work, worker queue inspection, a `claim-next` scheduler primitive, a receipt-backed work execution lifecycle for claim/start/complete/fail with duplicate suppression, retry backoff, and stale-receipt reclaim, plus a background scheduler loop exposed through the runtime loops surface for operator visibility. The local worker registry entry now refreshes live `current_load` and `queue_depth` during queue reads, claims, and receipt updates so placement consumes fresher capacity data. Not yet implemented are authority claim/handoff, richer scheduler decisions across many work types, broad multi-node execution, and rebalancing.
