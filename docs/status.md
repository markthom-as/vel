# Vel ‚Äî Current Status

What is implemented, what is partial, and what is next.

## Implemented

- **Capture storage** ‚Äî Insert captures, list by ID, lexical search (FTS5).
- **Artifacts API** ‚Äî Create and fetch artifacts; storage layer with metadata.
- **Run/event schema and inspection** ‚Äî `runs`, `run_events`, `events` tables; `vel runs` (with `--kind`, `--today`, `--limit`), `vel run inspect <id>`, `vel run status <id> <status>` (e.g. retry_scheduled, blocked); `GET /v1/runs?limit=&kind=&today=`, `GET /v1/runs/:id`, `PATCH /v1/runs/:id` (body: `{ "status": "..." }`).
- **Doctor diagnostics** ‚Äî `vel doctor` and `GET /v1/doctor` with structured checks (DiagnosticCheck + DiagnosticStatus).
- **Context endpoints (run-backed)** ‚Äî `GET /v1/context/today`, `GET /v1/context/morning`, `GET /v1/context/end-of-day` create a run, compute from orientation snapshot, write a managed **context_brief** artifact (write temp ‚Üí flush ‚Üí fsync ‚Üí rename; checksum, size_bytes, metadata_json with `generator`, `context_kind`, `snapshot_window`), link run ‚Üí artifact and artifact ‚Üí capture refs, append run events (`run_created`, `run_started`, `context_generated`, `artifact_written`, `refs_created`, `run_succeeded`), then transition to succeeded/failed.
- **Run timing** ‚Äî API and CLI expose `duration_ms` (derived from started_at/finished_at) on run list and run detail.
- **Runtime invariants** ‚Äî Documented in runtime-concepts: run_started precedes run_succeeded; artifact_written/refs_created before run_succeeded; run_failed does not produce artifact refs; run_succeeded implies artifact durability.
- **CLI** ‚Äî `vel health`, `vel capture`, `vel search`, `vel today`, `vel morning`, `vel end-of-day`, `vel doctor`, `vel inspect capture <id>`, `vel runs`, `vel run inspect <id>` (duration, artifact size e.g. 3.4KB, event timestamps in CLI).
- **Crate boundaries** ‚Äî Domain types (ContextCapture, SearchResult, OrientationSnapshot) live in `vel-core`; storage returns them; API layer maps to DTOs. `vel-storage` does not depend on `vel-api-types`.
- **Run events uniqueness** ‚Äî `(run_id, seq)` unique on `run_events`.
- **Typed run payloads** ‚Äî Run and RunEvent use `serde_json::Value` in domain/API; DB remains TEXT; (de)serialization at storage boundary.
- **Run transitions** ‚Äî Immutable transitions in `vel-core` (start/succeed/fail/cancel return new `Self`).
- **Service layer** ‚Äî Context generation and doctor logic live in `veld` services; routes are thin.
- **Artifact storage kind** ‚Äî `storage_kind` (managed | external) in schema, core, and API.
- **Global events (optional)** ‚Äî `DAEMON_STARTED` and `CONFIG_LOADED` emitted on veld startup; `SCHEMA_MIGRATION_COMPLETE` emitted after migrations run (with `schema_version` in payload). Role remains system audit log.
- **Run statuses (future use)** ‚Äî `RunStatus` includes `RetryScheduled` and `Blocked`; no transition logic yet (for future workflows).
- **Service-level result type** ‚Äî Context generation returns `ContextRunOutput<T>` (run_id, artifact_id, context_kind, data); routes map `.data` to API response.
- **Canonical doc names** ‚Äî Renamed `docs/vel-*.md` to `docs/*.md` (e.g. `runtime-concepts.md`, `data-model.md`); README and AGENTS.md updated.
- **Commitments (Phase A)** ‚Äî Schema (`commitments` table + indexes), domain type in `vel-core` (`Commitment`, `CommitmentId`, `CommitmentStatus`), storage CRUD, API (POST/GET /v1/commitments, GET/PATCH /v1/commitments/:id), CLI (`vel commitments`, `vel commitment add/done/cancel/inspect`). Capture promotion: captures with `capture_type == "todo"` auto-create an open commitment (source_type capture, commitment_kind todo). **Commitment dependency graph:** storage `insert_commitment_dependency`, `list_commitment_dependencies_by_parent/child`; API GET/POST /v1/commitments/:id/dependencies; CLI `vel commitment dependencies <id>`, `vel commitment add-dependency <parent> <child> [--type blocks]`. Spec: [docs/specs/commitments.md](specs/commitments.md).
- **Phase B ‚Äî Signal ingestion** ‚Äî `signals` table; storage `insert_signal`, `list_signals`; API POST/GET /v1/signals; adapters: calendar (ICS path/URL ‚Üí calendar_event signals), Todoist (snapshot JSON ‚Üí commitments + external_task signals), activity (`vel_invocation` via POST); CLI `vel sync calendar/todoist/activity`; config `calendar_ics_url`, `calendar_ics_path`, `todoist_snapshot_path`.
- **Phase C ‚Äî Inference** ‚Äî `inferred_state` table; service reads signals + commitments, computes morning state (inactive/awake_unstarted/underway/engaged/at_risk), meds_pending, prep_window_active; persists one inferred_state record; emits STATE_CHANGED.
- **Phase D ‚Äî Nudges** ‚Äî `nudges` table; nudge engine creates meds_not_logged, meeting_prep_window, morning_drift (gentle/warning level); API GET /v1/nudges, GET /v1/nudges/:id, GET/PATCH /v1/nudges/:id, POST /v1/nudges/:id/done, POST /v1/nudges/:id/snooze; CLI `vel nudges`, `vel nudge inspect <id>`, `vel nudge done <id>`, `vel nudge snooze <id> --minutes 10`.
- **Phase E ‚Äî Notification** ‚Äî Nudges surfaced via CLI (`vel nudges`) and API; desktop/watch stubbed for later.
- **Phase F ‚Äî Weekly synthesis** ‚Äî POST /v1/synthesis/week: run-backed (RunKind::Synthesis), writes **weekly_synthesis** artifact with canonical shape (week_start, week_end, summary, completed/open_commitment_ids, top_threads, drift_patterns, vel_self_review, suggested_adjustments); CLI `vel synthesize week`. Spec: [vel-weekly-synthesis-spec.md](specs/vel-weekly-synthesis-spec.md).
- **Phase G ‚Äî Observability** ‚Äî Events emitted: SIGNAL_INGESTED, STATE_CHANGED, NUDGE_GENERATED, NUDGE_RESOLVED; evaluate endpoint POST /v1/evaluate runs inference + nudge engine; CLI `vel evaluate`.
- **Schema (migrations 0012‚Äì0022)** ‚Äî `current_context` (singleton), `context_timeline`, `commitment_dependencies`, `commitment_risk`, `nudge_events`, `suggestions`, `threads`, `thread_links`, `vel_self_metrics`, `assistant_transcripts`; `signals.source_ref` added. See [docs/specs/vel-migrations-and-schema-spec.md](specs/vel-migrations-and-schema-spec.md).
- **Persistent current context** ‚Äî Inference engine writes canonical context (mode, morning_state, inferred_activity, next_commitment_id, prep/commute windows, meds_status, active_nudge_ids, top_risk_commitment_ids, global_risk_level/score) to `current_context`; material-change detection appends to `context_timeline`; GET /v1/context/current, GET /v1/context/timeline; CLI `vel context` (show), `vel context timeline`; `vel explain context`. Spec: [vel-current-context-spec.md](specs/vel-current-context-spec.md).
- **Nudge events** ‚Äî Append-only `nudge_events` on nudge create, done, snooze; supports explainability.
- **Explainability** ‚Äî GET /v1/explain/nudge/:id (nudge + inference/signals snapshot); GET /v1/explain/context; GET /v1/explain/commitment/:id (commitment + risk snapshot + why in context); GET /v1/explain/drift (attention_state, drift_type, reasons, signals_used, commitments_used); CLI `vel explain nudge <id>`, `vel explain context`, `vel explain commitment <id>`, `vel explain drift` (all support `--json`).
- **Policy engine** ‚Äî Resolution policies run first (auto-resolve meds/prep/drift nudges when conditions no longer hold); each nudge creation stores structured explanation in `metadata_json` (policy, decision, level, reasons). Spec: [vel-policy-engine-spec.md](specs/vel-policy-engine-spec.md).
- **Signal adapters** ‚Äî On capture create, a `capture_created` signal (source `vel`) is inserted in addition to CAPTURE_CREATED event; POST /v1/signals accepts `vel_feedback` for user feedback. Spec: [vel-signal-adapter-spec.md](specs/vel-signal-adapter-spec.md).
- **Policy config** ‚Äî Loaded from `config/policies.yaml` at startup (or `VEL_POLICIES_PATH`); missing/malformed file fails startup. Injected into nudge engine (default_prep_minutes, commute_leave_time thresholds). Spec: [vel-agent-next-steps-policy-config-commute.md](specs/vel-agent-next-steps-policy-config-commute.md).
- **commute_leave_time nudge** ‚Äî Fires only when calendar event has `travel_minutes`; leave_by = event_start - travel_minutes; escalation ladder (gentle/warning/danger) from config; resolution on event start or commitment resolve.
- **Context explain** ‚Äî GET /v1/explain/context returns `signals_used`, `commitments_used`, `risk_used`, `reasons`; inference stores used ids in current context JSON.
- **Thread graph** ‚Äî Storage: insert_thread, get_thread_by_id, list_threads, update_thread_status, insert_thread_link, list_thread_links. API: GET/POST /v1/threads, GET/PATCH /v1/threads/:id, POST /v1/threads/:id/links. CLI: `vel threads`, `vel thread inspect/close/reopen`. Spec: [vel-thread-graph-spec.md](specs/vel-thread-graph-spec.md).
- **Risk engine compute** ‚Äî Consequence, proximity, dependency pressure (no uncertainty/progress-penalty in first version). POST /v1/evaluate runs inference ‚Üí risk::run ‚Üí nudge engine; commitment_risk table populated; GET /v1/risk (compute and return), GET /v1/risk/:id; CLI `vel risk`, `vel risk <commitment_id>`. Spec: [vel-risk-engine-spec.md](specs/vel-risk-engine-spec.md), [vel-agent-next-implementation-steps.md](specs/vel-agent-next-implementation-steps.md).
- **Attention/drift in current context** ‚Äî Inference writes `attention_state`, `drift_type`, `drift_severity`, `attention_confidence`, `attention_reasons` into context JSON (morning_drift, prep_drift heuristics). Spec: [vel-attention-and-drift-detection-spec.md](specs/vel-attention-and-drift-detection-spec.md).
- **Suggestions (steering loop)** ‚Äî Storage: insert_suggestion, list_suggestions, get_suggestion_by_id, update_suggestion_state. API: GET /v1/suggestions, GET/PATCH /v1/suggestions/:id. CLI: `vel suggestions`, `vel suggestion inspect/accept/reject/modify`. Two types (increase_commute_buffer, increase_prep_window) to be triggered by repeated evidence; scaffolding in place. Spec: [vel-agent-next-implementation-steps.md](specs/vel-agent-next-implementation-steps.md).
- **Project synthesis** ‚Äî POST /v1/synthesis/project/:slug; run_project_synthesis filters commitments by project, writes **project_synthesis** artifact (open_commitments, active_threads, completed_commitment_ids, etc.); CLI `vel synthesize project <name>`. Spec: [vel-agent-next-implementation-steps.md](specs/vel-agent-next-implementation-steps.md).
- **Integration tests** ‚Äî Commute policy: no commute nudge when calendar event has no travel_minutes; commute nudge fires with travel_minutes in leave window; context explain returns signals_used, commitments_used, reasons; resolution order: resolved nudge stays resolved after second evaluate. **Canonical day fixture:** shared fixture (Meeting with Dimitri, prep 30, travel 40, meds/prep/commute commitments); ¬ß6.1 context assertions (prep/commute window, meds_status, mode); ¬ß6.2 risk assertions (non-empty risk list); ¬ß6.3 nudge snooze suppresses until snoozed_until, event start suppresses commute nudge; ¬ß6.5 explain references commitments and signals; ¬ß6.6 project synthesis artifact; Variant A (success: meds done removes meds nudge); Variant B (drift: commute danger + drift/attention in context); Variant C (suggestion: repeated commute danger triggers increase_commute_buffer).
- **Feedback compliance (review round)** ‚Äî Evaluate order: risk ‚Üí inference ‚Üí nudges so context incorporates risk snapshot; inference reads `commitment_risk` for `top_risk_commitment_ids`, `risk_used`, `global_risk_level`/`score`; risk factors include `dependency_ids`; policy engine uses risk for meeting_prep_window level (warning when meeting_risk >= high); suggestion triggers on repeated evidence (commute danger, prep resolved) via `suggestions::evaluate_after_nudges` after evaluate; observability: `vel explain context`, `vel explain commitment <id>`, `vel explain drift`. Events table present (migration 0004). No new subsystems added.
- **Latest repo feedback** ‚Äî [vel-latest-repo-feedback.md](reviews/vel-latest-repo-feedback.md). **Explain routes read-only:** `explain_commitment` no longer calls `risk::run()`; it reads latest persisted risk via `list_commitment_risk_recent(id, 1)`. **First event = next future event:** inference uses the nearest upcoming event at or after `now`, with fallback to most recent past event for ‚Äúcurrently active‚Äù display. **Commute nudge resolution:** resolved only when event has started or the related commitment is resolved (no global `open_commitments.is_empty()`).
- **Architecture boundaries (detailed next steps)** ‚Äî Nudge engine refactored to consume **current context + risk + nudges + policy** only: no recomputation of meds/prep/morning from raw signals. It reads `get_current_context()`, `list_commitment_risk_latest_all()`, `list_nudges()`, policy config; uses context fields `meds_status`, `prep_window_active`, `commute_window_active`, `morning_state`, `leave_by_ts`, `next_event_start_ts` for resolution and creation. Inference writes `leave_by_ts` and `next_event_start_ts` into context so commute nudge level can be computed from context. Single source of truth for present state is context reducer (inference).
