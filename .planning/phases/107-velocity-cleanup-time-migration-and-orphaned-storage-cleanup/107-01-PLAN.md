# Phase 107 Plan

## Objective

Finish the time-library migration and make the storage cleanup truthful by distinguishing active/planned schema from genuinely orphaned tables.

## Deliverables

- [ ] active `veld` services/tests targeted by the audit use `time` instead of `chrono`
- [ ] timezone-aware behavior is preserved through a `time`-compatible solution
- [ ] every table flagged by the audit is reclassified as either active, planned-but-kept, or truly orphaned
- [ ] only truly orphaned schema is dropped, and any kept schema has its intended owner/use documented
- [ ] workspace dependencies no longer carry stale temporal overlap once the migration is complete

## Baseline Checks

1. Confirm current temporal overlap and compile surface:
   - `rg -n 'chrono|chrono_tz' crates/veld Cargo.toml`
   - `cargo test -p veld phase64_recurrence_and_availability -- --nocapture`
2. Reconfirm schema usage before any migration planning:
   - `rg -n 'vel_self_metrics|storage_targets|verification_records' crates migrations docs .planning`
3. Record the current backup/trust ownership evidence:
   - `crates/vel-storage/src/repositories/backup_runs_repo.rs`
   - `docs/future/hosted-v1-signalling-backup-dashboard-spec.md`

## Execution Checklist

1. Migrate `crates/veld/src/services/timezone.rs` first and establish the replacement timezone approach.
2. Migrate the remaining flagged service files (`recurrence_materialization`, `availability_projection`, `now`, `planning_profile`) and their tests incrementally.
3. Remove `chrono` / `chrono-tz` from the workspace only after the last active use is gone and tests pass.
4. Reclassify the flagged schema:
   - `storage_targets` → active backup foundation, keep
   - `verification_records` → preserve unless proven unrelated to backup/trust verification
   - `vel_self_metrics` → preserve unless proven unrelated to self-awareness or reflective tuning direction
5. If any table is truly unowned, add a forward drop migration and document why it is safe.
6. If a table is preserved, add or update nearby docs/planning notes so the intended owner is explicit instead of implied.

## Sequencing Notes

- do the time migration before dependency cleanup so the code tells you when `chrono` is truly gone
- do schema re-verification before writing any drop migration; the audit already had one false positive (`storage_targets`)
- prefer documenting preserved schema over deleting it speculatively and rebuilding it later under a new name

## Exit Criteria

- `chrono` is no longer part of the active `veld` service/test path targeted by the audit
- no table remains in the “probably orphaned” state; each flagged table is either dropped or explicitly justified
- backup/trust and self-awareness directions stay coherent with the Master Plan
