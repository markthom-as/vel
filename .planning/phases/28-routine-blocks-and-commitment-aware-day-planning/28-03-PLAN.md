# 28-03 Plan

## Slice

Embody planned-day output in `Now`, `Threads`, and `Settings` without creating a shell-owned planner.

## Why this slice third

Once the backend emits a real plan, the operator needs to see it in the existing product surfaces without letting the UI become the source of planning truth.

## Outcomes

- `Now` shows compact day-plan shape and immediate judgment/defer pressure
- `Threads` remains the escalation lane for longer planning disagreement or manual shaping
- `Settings` surfaces summary-first planning/recovery posture without becoming a second planner destination
- shells consume typed planner output directly instead of deriving their own scheduling semantics

## Likely Touch Points

- `clients/web/src/components/NowView.tsx`
- `clients/web/src/components/ThreadView.tsx`
- `clients/web/src/components/SettingsPage.tsx`
- typed web/API boundaries if the planner output widens existing payloads

## Verification

- focused web/component tests for planner rendering and escalation behavior
- contract round-trip tests for any widened DTOs
- truth checks that shell copy matches the bounded backend-owned planning model
