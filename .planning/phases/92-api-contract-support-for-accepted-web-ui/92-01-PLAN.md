# Phase 92 Plan

## Objective

Add the backend/API contract work required to make the accepted web UI behaviors durable, editable, and truthful.

## Scope

### In scope

- extend DTOs and request payloads where accepted UI behaviors cross the transport boundary
- add or refine web-facing routes/mutations for accepted task-lane, assistant-entry, and preference behavior
- keep transport changes typed and documented
- update frontend boundary code in the same slice as backend DTO changes

### Out of scope

- speculative backend redesign beyond accepted UI needs
- new surfaces or new providers
- visual polish tasks that do not require API support

## Work Buckets

1. Assistant entry contract
   - preserve the now-implemented explicit manual intent + typed attachment seam
   - bind the durable `follow_up` handle more deeply into accepted `Now` affordances instead of leaving it as a passive DTO

2. `Now` transport and mutation support
   - remove artificial lane truncation from the transport used by the accepted UI
   - support multiple active tasks and explicit lane membership
   - add a mutation for lane moves / reorder where needed
   - expose a resolved header block for `CLIENT | LOCATION`, time/context, and related precedence

3. Thread list metadata
   - preserve the now-implemented optional project label, message count, and explicit row freshness fields on conversation rows
   - add any remaining participant-summary fields separately rather than overloading the row DTO

4. Operator preferences
   - add typed read/write settings for the `System` appearance/accessibility controls that are currently local-only

5. Capability/trust surfacing
   - decide the minimal truthful seam for browser voice capability degradation if it must appear in shared nudges/system trust

## Contract-Ready Delta List

`System` settings/config mutations are broader than the other seams in this phase, so the implementation should be split in smallest-first order:

1. Define typed operator-preference DTOs for the editable `System` fields only.
   - target: `crates/vel-api-types/src/lib.rs`
   - include only the fields that the current web surface can truthfully persist
   - keep inspect-only settings out of the write contract

2. Narrow the `/api/settings` patch surface to explicit typed sections.
   - target: `crates/veld/src/routes/chat.rs`
   - map typed request fields into existing storage keys and service helpers
   - preserve the current read payload shape until the client boundary is updated

3. Add round-trip contract tests for the typed settings mutation.
   - target: `crates/veld/tests/chat_settings.rs` or the closest existing settings test module
   - verify persisted values come back through `GET /api/settings`
   - verify invalid values reject cleanly at the boundary

4. Update the web boundary after the backend contract is stable.
   - target: `clients/web/src/types.ts`
   - target: the settings data layer that actually calls `PATCH /api/settings`
   - keep this separate from the view rewrite so the seam stays truth-first

## Success Criteria

1. Accepted UI behaviors that require backend truth no longer depend on fake client-local state.
2. DTO/request changes are updated in both Rust transport types and web boundary code in the same slice.
3. Existing capabilities such as conversation archive and commitment complete/reopen are reused rather than duplicated.
4. The follow-on frontend revision can consume these seams without inventing new shadow state.
