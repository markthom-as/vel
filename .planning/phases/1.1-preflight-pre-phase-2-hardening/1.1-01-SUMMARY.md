# Phase 1.1 Plan 01: Preflight Hardening Summary

**One-liner:** SQLite WAL mode via guarded sqlx API, tracing::warn! on integration settings corruption, and auth middleware extracted to `crates/veld/src/middleware/mod.rs` — plus full workspace clippy clean.

## What Was Built

### 1. SQLite WAL Mode (SC-2)
**File:** `crates/vel-storage/src/infra.rs`

Added `SqliteJournalMode::Wal` to `sqlite_connect_options` via the typed sqlx API, guarded for file-backed databases only:

```rust
use sqlx::sqlite::{SqliteConnectOptions, SqliteJournalMode};

let options = if db_path != ":memory:" {
    options.journal_mode(SqliteJournalMode::Wal)
} else {
    options
};
```

Two new tests confirm the behavior:
- `wal_mode_enabled_for_file_db`: queries `PRAGMA journal_mode`, asserts `"wal"` (RED → GREEN)
- `wal_mode_skipped_for_memory_db`: confirms in-memory pool connects cleanly (guard test)

### 2. Integration Settings Corruption Hardening (SC-1)
**Files:** `crates/veld/src/services/integrations.rs`, `crates/veld/src/services/integrations_todoist.rs`

Replaced the silent `unwrap_or_default()` on deserialization failure with an observable variant that emits `tracing::warn!`:

```rust
.map(|value| {
    serde_json::from_value::<T>(value).unwrap_or_else(|err| {
        tracing::warn!(
            key = %key,
            error = %err,
            "integration settings deserialization failed, using defaults"
        );
        T::default()
    })
})
```

Two regression tests guard the safe fallback contract:
- `load_todoist_settings_returns_default_when_key_missing`
- `load_todoist_settings_returns_default_when_json_corrupt`

### 3. Auth Middleware Extraction (SC-3)
**New file:** `crates/veld/src/middleware/mod.rs`
**Modified:** `crates/veld/src/app.rs`, `crates/veld/src/lib.rs`, `crates/veld/src/main.rs`

Extracted ~120 lines of auth logic from `app.rs` to a dedicated module:
- `RouteExposureClass`, `HttpExposurePolicy`, `ExposureGate` (all `pub(crate)` with `pub(crate)` fields)
- `enforce_exposure_gate`, `extract_bearer_token`, `extract_header_token`, `expected_token_for_class`
- `unauthorized_response`, `env_flag_enabled`
- `OPERATOR_AUTH_HEADER`, `WORKER_AUTH_HEADER`

`app.rs` now contains only route builders, imports, and the test module. The 160 existing auth tests pass via `use super::*` + explicit `use crate::middleware::{OPERATOR_AUTH_HEADER, WORKER_AUTH_HEADER}` in the test module.

Key discovery: `main.rs` has its own module tree separate from `lib.rs` and required an explicit `mod middleware;` declaration.

## Decisions Made

1. **Re-export scope minimized**: Only `OPERATOR_AUTH_HEADER` and `WORKER_AUTH_HEADER` are imported in the test module explicitly from `crate::middleware`. The other auth symbols (`HttpExposurePolicy`, `RouteExposureClass`, etc.) reach tests via `use super::*` pulling the private imports from the parent module's scope.

2. **main.rs module declaration**: Rust binary targets (`main.rs`) have independent module trees from library targets (`lib.rs`). Adding `mod middleware;` to `lib.rs` alone was insufficient — `main.rs` needed its own `mod middleware;` declaration.

3. **Lifetime elision in middleware**: `expected_token_for_class` had an explicit `'a` lifetime that clippy correctly identified as electable (single input reference). Fixed to use anonymous lifetime.
