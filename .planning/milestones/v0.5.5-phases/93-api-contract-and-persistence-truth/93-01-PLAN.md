# Phase 93 Plan

## Objective

Make the accepted web behavior truthful at the backend/API boundary before further functionality and polish work.

## Scope

### In scope

- widen the `Now` lane contract so the web client stops depending on fake local lane semantics
- add or tighten persisted settings/config seams used by `System`
- keep Rust DTOs and web boundary code in sync in the same slice
- add focused backend and boundary verification for changed contracts

### Out of scope

- browser-level visual polish
- speculative backend redesign beyond accepted UI needs
- unrelated provider expansion
- acceptance closeout

## Implementation slice

1. Materialize the true remaining API scope in planning and remove stale assumptions.
2. Tighten persisted `System` settings so the current surface can distinguish truthful edits from inspect-only fields.
3. Widen the `Now` task-lane contract to support explicit lane membership and durable multi-active semantics without frontend shadow state.
4. Update the web boundary in the same slice and keep route handlers thin.
5. Add focused Rust and web-boundary verification for the new seams.

## Verification

- focused Rust route/service tests for changed settings and `now` contracts
- targeted web boundary decode/data-layer tests where DTO shapes change
- direct runtime checks where cheap and available

## Success Criteria

1. `System` no longer implies persisted edits for fields the backend cannot actually save.
2. `Now` lane semantics are transport-backed instead of primarily client-local.
3. Rust DTOs and web boundary code remain aligned in one slice.
4. Phase 94 can bind behavior to real persisted/runtime state rather than further approximation.
