# Phase 106 Plan

## Objective

Simplify the active cleanup seams identified in the audit by removing unnecessary abstraction, deleting obviously dead policy scaffolding, and turning blanket warning suppression into a more truthful, reviewable posture.

## Deliverables

- [ ] `CapabilityResolver` is replaced with a direct module-level function and its tests/callers still prove deterministic behavior
- [ ] `ToolRunner` is replaced with a direct async function and its tests/callers still prove success/timeout behavior
- [ ] dead policy structs and accessors are removed from `policy_config.rs` while preserving `commute_leave_time` and any actively used config surface
- [ ] crate-level `#![allow(dead_code)]` usage is reduced for the touched crates, with any unavoidable leftovers moved to narrow, documented item-level allows
- [ ] warning/dead-code follow-on debt is documented truthfully instead of hidden behind blanket suppression

## Baseline Checks

1. Run the narrowest relevant Rust tests for the touched seams before editing:
   - `cargo test -p veld capability_resolver`
   - `cargo test -p veld tool_runner`
   - `cargo test -p vel-cli policy_check`
2. Run a warning baseline:
   - `cargo check -p veld --all-targets`
   - `cargo check -p vel-cli --all-targets`

## Execution Checklist

1. Convert `crates/veld/src/services/capability_resolver.rs` from a trait-plus-unit-struct pattern to a plain helper function.
2. Update `crates/veld/src/services/command_lang.rs` imports and call sites to use the new capability helper directly.
3. Convert `crates/veld/src/services/tool_runner.rs` from a trait-plus-unit-struct pattern to a plain async helper function.
4. Update `command_lang.rs` to call the new tool-runner helper directly and keep test coverage on success/timeout behavior.
5. Remove dead policy structs, defaults, fields, and accessors from `crates/veld/src/policy_config.rs`, preserving only the config that is actually referenced.
6. Re-run search and compile checks to surface remaining dead-code pressure in the touched crates.
7. Replace blanket `dead_code` suppression only where the touched code now compiles cleanly; use targeted `#[allow(dead_code)]` with a reason comment for intentionally staged surfaces that cannot be activated in-scope.
8. Refresh `docs/DEAD-CODE.md` or a nearby planning artifact if the warning-debt posture materially changes.

## Sequencing Notes

- Do the trait removals first because they are small, low-risk, and tighten the active command-language seam immediately.
- Do the policy cleanup second because it is isolated and should not interact with the trait simplifications.
- Do suppression cleanup last so warning output reflects the post-cleanup code shape instead of the pre-cleanup one.
- Keep each sub-slice reviewable; if suppression cleanup mushrooms, stop at the nearest honest boundary and document the exact deferred cluster instead of hiding it again.

## Exit Criteria

- the active command-language cleanup seams no longer use fake polymorphism
- `policy_config.rs` no longer advertises inactive policy surfaces as if they were live
- touched crates move toward truthful warning visibility without reintroducing noisy blanket suppression
- any remaining dead-code debt after this phase is named and localized
