# Phase 109 Plan

## Objective

Break up the largest navigability hotspots in the daemon and transport crates while preserving existing runtime behavior and consumer-facing API shape.

## Deliverables

- [ ] `crates/veld/src/app.rs` is reduced to a clear assembly root with route grouping and test ownership separated
- [ ] `crates/veld/src/services/now.rs` is split into submodules that reflect extraction/build/composition/output responsibilities
- [ ] `crates/vel-api-types/src/lib.rs` becomes a re-export index over grouped DTO modules
- [x] consumer imports continue to work without broad call-site churn
- [ ] tests/build checks demonstrate that the modularization is behavior-preserving

## Baseline Checks

1. Capture the current large-file seams and public consumers:
   - `rg -n 'use vel_api_types::' crates clients`
   - `rg -n 'build_app|public_routes|operator_authenticated_routes|worker_authenticated_routes' crates/veld/src`
   - `rg -n 'Now' crates/veld/src/services/now.rs`
2. Run the narrowest meaningful automated baseline:
   - targeted `veld` route tests
   - targeted `Now` service tests if present
   - at least one consumer compile path for `vel-api-types`

## Execution Checklist

1. Split `app.rs` by route grouping / error glue / test ownership while keeping the top-level app builder obvious.
2. Split `now.rs` into stable submodules and keep the existing public entrypoint intact.
3. Split `vel-api-types/src/lib.rs` into grouped modules with `pub use` re-exports from `lib.rs`.
4. Keep module boundaries aligned with existing conceptual seams so future changes land in obvious files.
5. Only take minimal adjacent cleanup when it directly improves the split and stays easy to review.
6. Update nearby docs/comments if the new layout changes the obvious owner file for future work.

## Suggested Sequencing

1. `vel-api-types` first if the re-export pattern is straightforward and low-risk
2. `app.rs` second because route grouping is largely assembly work
3. `now.rs` last because it has the highest domain density and the greatest risk of accidental logic churn
