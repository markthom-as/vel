# Phase 108 Plan

## Objective

Choose and implement one defensible forward error-handling pattern, then make `vel-sim` and `vel-agent-sdk` first-class existing-surface integrations instead of retained but hidden crates.

## Deliverables

- [ ] one concrete route → service → storage seam uses the chosen forward error-handling pattern and nearby docs explain the rule
- [ ] `vel-sim` has an explicit integration point under an existing CLI surface, most likely `vel evaluate` or a closely related existing command family
- [ ] `vel-agent-sdk` has an explicit integration point under the supervised execution/runtime lane, most likely `vel exec` and related docs
- [ ] docs explain what each crate is for, when operators/developers should use it, and what remains future work
- [ ] any unconverted error-boundary areas are named explicitly as follow-on work rather than left as generic inconsistency debt

## Baseline Checks

1. Review the current error pattern split in a real seam before changing anything:
   - route returns `AppError`
   - service returns service-specific error
   - storage returns `sqlx::Error`/storage-specific error
2. Reconfirm current crate consumers:
   - `vel-sim` via `veld-evals`
   - `vel-agent-sdk` via protocol tests and supervised-execution research/docs
3. Run focused baseline checks:
   - `cargo test -p veld agent_sdk -- --nocapture`
   - `cargo test -p vel-sim -- --nocapture`
   - `cargo test -p veld-evals -- --nocapture`

## Execution Checklist

1. Select one representative seam for error normalization and document why it is the right proving path.
2. Apply a single forward pattern to that seam and update nearby docs/comments so future work can extend the same rule.
3. Extend an existing CLI surface to make `vel-sim` useful without creating a new top-level command family.
4. Extend an existing supervised-execution/runtime CLI surface to make `vel-agent-sdk` useful without creating a new top-level command family.
5. Update operator/developer docs so the retained crates have explicit purpose and usage instead of only historical references.
6. Record any still-deferred normalization or SDK growth as specific follow-ons.

## Suggested Integration Shape

- `vel-sim`: attach to `vel evaluate` or its adjacent evaluation workflow as a deterministic replay/proof seam
- `vel-agent-sdk`: attach to `vel exec` / supervised execution docs as the reference client for external or repo-local runtime envelopes

## Sequencing Notes

- decide the error pattern before touching multiple layers so the crate integration work does not inherit more inconsistency
- land the crate integrations through docs plus CLI affordances together; “keep” without an entry point is not enough
- keep the integrations dev/operator-oriented for now rather than widening user-facing top-level command taxonomy

## Exit Criteria

- a reviewer can point to one concrete forward error-handling pattern and one real implementation seam
- `vel-sim` and `vel-agent-sdk` are obviously alive and reachable from current workflows
- the retained crates have documented jobs that line up with the Master Plan
