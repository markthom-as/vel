# `0.5` Field Ownership Matrix

**Captured:** 2026-03-22  
**Status:** Locked implementation-precondition artifact for milestone `0.5`

## Purpose

This matrix turns ownership policy from tribal lore into checked-in contract.

It is the required pre-implementation authority for source-owned precedence, merge behavior, outward write eligibility, and conflict handling across the proving adapters.

If implementation wants to widen or refine a field rule, the matrix must be updated deliberately rather than allowing ownership semantics to drift into handler code.

## Matrix Columns

Each row in the provider-specific matrices must define:

- field
- canonical owner
- provider owner
- merge rule
- conflict rule
- writable from Vel
- outward write target
- notes / caveats

## Required Matrices

`0.5` requires these concrete matrices before implementation is considered ready:

1. Todoist task matrix
2. Google event matrix
3. attendee/person identity ownership matrix where identity promotion or provider-stub handling matters

## Ownership Classes

Use only these ownership classes unless the packet is explicitly revised:

- `canonical`
- `provider`
- `shared_mapped`
- `derived_read_model`
- `runtime_control`

## Default Law

- source-owned provider fields win on sync conflicts
- shared mapped fields still require explicit ownership and merge rules per row
- derived read models are not authoritative write targets
- runtime/control records do not backdoor canonical ownership
- outward writes are legal only when the row explicitly allows them and the membrane/policy path permits them

The attendee/person identity matrix must at minimum cover:

- cross-provider corroboration

Milestone verification must prove at least:

- one field-ownership conflict case per adapter
