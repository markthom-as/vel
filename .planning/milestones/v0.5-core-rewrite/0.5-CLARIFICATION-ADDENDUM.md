# `0.5` Clarification Addendum

**Captured:** 2026-03-22  
**Status:** Locked clarification layer for milestone `0.5`

## Purpose

This addendum locks implementation-critical clarifications that were still too easy to "discover" accidentally in code.

These clarifications are packet law for milestone `0.5`.

They refine, but do not reopen, the core authority established across Phases 57 through 65.

## 1. Canonical Task Status

The canonical `TaskStatus` enum is:

- `inbox`
- `ready`
- `in_progress`
- `blocked`
- `done`
- `canceled`

### Mapping posture

Todoist does not define the canonical ontology.

Todoist and future providers map into this Vel-owned status set and preserve provider-native raw status in provider facets when needed.

### Mapping guidance

- untriaged or capture-like provider state -> `inbox`
- open/default actionable state -> `ready`
- explicitly started or locally active focus state -> `in_progress`
- dependency/wait/policy hold state -> `blocked`
- completed -> `done`
- intentionally abandoned/canceled -> `canceled`

## 2. Canonical Task Fields

These task fields are first-class canonical fields in `0.5`:

- `title`
- `description`
- `status`
- `priority`
- `due`
- `task_type`
- `parent_task_ref`
- `tags`
- `created_at`
- `updated_at`
- `completed_at`
- `source_refs`
- `provider_facets`

### Canonical due shape

Use a canonical due structure with at least:

- `kind`: `date | floating_datetime | zoned_datetime`
- `value`
- `timezone` for `zoned_datetime`
- `is_recurring_snapshot` where imported provider state requires it

### Canonical priority shape

The canonical stored priority enum is Vel-owned and semantic:

- `critical`
- `high`
- `medium`
- `low`
- `lowest`

Provider-native and operator-facing shorthand maps into this scale:

- `p0 -> critical`
- `p1 -> high`
- `p2 -> medium`
- `p3 -> low`
- `p4 -> lowest`

### Canonical subtasks

Subtasks are modeled canonically through parent/child relationships:

- `parent_task_ref`

Subtasks are not a separate ad hoc provider shape.
`0.5` supports single-parent hierarchy only.

### Canonical task type

`task_type` is a first-class canonical field in `0.5`.

The initial enum is:

- `generic`
- `maintain`
- `practice`
- `ritual`
- `chore`

Do not widen this taxonomy casually during `0.5`.

### Task system posture

Vel should be able to become the primary task system rather than remain a thin Todoist mirror.

That means due, priority, description, subtasks, tags, and typed task semantics all remain canonical Vel concerns even when Todoist is the first major shaping adapter.

## 3. Task Semantics And Tags

Vel supports both:

- canonical raw tags
- canonical structured task semantics

This is not an either/or decision.

### Canonical rule

Raw tags remain canonical and preserved for:

- user fluency
- round-trip safety
- provider sync
- future view/query ergonomics

Structured task semantics are also canonical and may be populated manually, by workflow/runtime, or by tag interpretation.

### Initial `task_semantics` shape

The initial typed home for interpreted task meaning may include:

- `estimated_duration_minutes`
- `time_of_day_hint`
- `routine_block_ref`
- `energy_level_hint`
- `context_tags`
- `schedule_flexibility`

These fields are optional in `0.5`, but the typed home must exist so task planning meaning does not remain trapped in raw tag conventions forever.

Do not store a separate `is_routine` boolean in `0.5`.
Use `task_type` and `routine_block_ref` instead.

### Interpretation rule

Vel may interpret recognized tags into structured task semantics.

That interpretation must:

- preserve raw tags
- preserve provenance
- avoid destructive conversion

Recommended provenance values include:

- `manual`
- `inferred_from_tag`
- `workflow`
- `provider_sync`

### Outward sync classes

No `task_semantics` field syncs outward unless there is an explicit mapping rule.

Allowed to sync outward to Todoist labels when explicitly configured:

- `time_of_day_hint`
- `estimated_duration_minutes`
- `energy_level_hint`
- `context_tags`
- `routine_block_ref` only through an explicit configured label convention

Interpretable but outward only through explicit future mapping config:

- `schedule_flexibility`
- `focus_mode_hint`
- future derived behavioral classifications

Vel-only:

- confidence or provenance fields
- inferred behavioral analytics
- clustering or duplicate signals
- internal workflow routing hints not intended for provider round-trip

### Tag namespace posture

`0.5` tolerates freeform tags and supports configurable interpretation rules.

It also reserves a recommended namespaced convention:

- `time:morning`
- `duration:15m`
- `energy:low`
- `routine:admin`
- `focus:deep`
- `context:errands`

## 4. Todoist Provider Facet And Task Mapping

Todoist sections remain non-first-class in `0.5`, but section metadata must still have a typed provider-facet home.

### Recommended provider facet posture

Use a typed `TodoistTaskFacet` provider shape such as:

- `project_id`
- `section_id`
- `section_name_snapshot`
- `parent_task_id`
- `labels`
- `priority`
- `due`
- `is_deleted_upstream`
- other adapter-owned provider-native metadata

### Todoist mapping law

- due -> canonical due
- priority -> canonical priority
- description -> canonical description
- subtasks -> canonical parent/child relation
- labels -> canonical tags plus provider facets
- sections -> typed provider facet, not first-class canonical object

### Tag interpretation law

Todoist labels/tags sync into canonical tags.

Vel may interpret selected tags into structured `task_semantics`, but:

- raw tags remain preserved
- interpreted semantics preserve provenance
- provider-native label behavior remains available in provider facets
- tag interpretation must not silently destroy round-trip fidelity

## 5. Attached Comments

`AttachedCommentRecord` is a shared cross-provider core contract in `0.5`, with provider-owned extensions in provider facets.

### Minimum core shape

- `id`
- `parent_object_ref`
- `body`
- `author_ref` or provider author stub
- `created_at`
- `updated_at`
- `source_refs`
- `visibility_or_scope` where later policy surfaces need it
- `provider_facets`

### Rule

Todoist comments do not become canonical `Message` objects by default.

Future providers may reuse the same `AttachedCommentRecord` core contract with their own provider facets.

## 5a. Eventful Provider Integration Doctrine

Providers may expose both:

- current object state
- historical activity or change signals

Vel must preserve this distinction.

For Todoist in particular, `0.5` should be shaped to support:

- initial full import of current state
- incremental current-state sync using provider cursor or token semantics where available
- ingestion of provider activity or history signals where available
- persistence of normalized provider event records separate from canonical task state
- deterministic projection from provider event history into analytics or behavior views
- preserved raw event payload snapshots where available

### Normalized history type

The normalized task-side history type in `0.5` is `TaskEvent`.

`TaskEvent` covers:

- provider-originated changes
- sync-diff-inferred changes
- local Vel-originated writes or intents
- workflow changes affecting canonical task behavior history

### Canonical rule

Current canonical task state, sync/linkage state, and provider event history are three distinct layers:

1. canonical task state
2. sync and linkage state
3. append-only provider event or activity history

Do not collapse these into one store or one object shape.

### Minimum Todoist history signals to preserve

Where observable through provider history, sync deltas, or local write intent trails, `0.5` should preserve at minimum:

- task created
- task completed
- task reopened
- task deleted or restored
- due changed / rescheduled
- priority changed
- title/content changed
- description changed
- labels/tags changed
- project or section moved
- parent/subtask relation changed

### Provenance rule

Provider history records must preserve how the event was observed, such as:

- provider-explicit activity event
- sync-diff-inferred change
- local `WriteIntent` or workflow-caused mutation

Recommended `TaskEvent` provenance values include:

- `provider_event`
- `sync_diff_inferred`
- `local_write_intent`
- `local_write_applied`
- `workflow_action`

Behavior analytics may be derived from this history, but must not replace it.

Normalized history is the primary query surface.
Raw payload preservation is the forensic and provenance substrate.

### Derived analytics minimum

`0.5` should derive at minimum, where the event model supports it cheaply and correctly:

- `reschedule_count`
- `push_later_count`
- `rewrite_count_title`
- `rewrite_count_description`
- `rewrite_count_combined`

Record all due changes broadly in history first.
Compute `push_later` or `pull_forward` analytics as derived views, not as narrowed ingest rules.

### Why this matters

This history layer is required to support questions such as:

- how often a task was rescheduled
- how often similar tasks were manually rewritten
- repeated avoidance or churn patterns
- completion latency and task half-life by tag, project, or task semantics

### Local-task rule

Local-only Vel tasks are fully editable without provider-mapping constraints.

Provider mapping constraints only apply when projecting to or reconciling with an external provider.

## 6. Google Attendee Identity And Stub Promotion

Attendee identity follows this rule:

- normalized email first
- provider-scoped stub when no stable canonical identity is available

Provider-scoped attendee stubs may exist without email when only display metadata is available.
Such stubs must be marked low-confidence and non-auto-promotable.

### Promotion rule

A provider-scoped attendee stub promotes to canonical `Person` when at least one of these is true:

- explicit user confirmation
- repeated stable email with sufficient merge confidence
- corroborating identity from another provider/source
- system policy allows email-based person creation

### Promotion posture

Promotion is an in-place logical upgrade, not a destructive replacement.

It must preserve:

- stable internal reference where possible
- provider/source provenance
- provider account provenance
- first-seen timestamp
- promotion or merge reason
- prior stub identifier as historical alias/reference

### Canonical attendee fields

Canonical attendee participation fields in `0.5` include:

- `response_status`
- `is_organizer`
- `is_self`
- `is_optional`
- `is_resource`

## 7. Google Recurrence Minimum Supported Write Scope

Supported in `0.5`:

- non-recurring event update
- whole-series update
- bounded single-occurrence override/update
- bounded single-occurrence delete/cancel
- whole-series delete

Explicitly unsupported in `0.5`:

- `this_and_following`

### Fidelity rule

Recurrence fidelity means:

- canonical series object preserved
- derived/materialized occurrences supported
- exceptions/overrides preserved where needed
- provider recurrence metadata retained as facet where native model does not fully absorb it
- no silent flattening
- no fake round-trip promises for unsupported scopes

### Bounded occurrence-write rule

Single-occurrence support in `0.5` is intentionally bounded.

It is only for explicit cases such as:

- changing one occurrence's time
- changing one occurrence's title
- changing one occurrence's location
- changing one occurrence's description
- changing one occurrence's attendance where provider support is clear
- canceling one occurrence

`0.5` does not promise generalized recurrence surgery.

### Override materialization rule

Recurring override objects are only materialized when:

- observed from provider import or sync
- changed locally/by Vel

Do not eagerly pre-materialize all future occurrences as override records.

Canceled recurring occurrences should be queryable as explicit canceled overrides, not merely implicit absences.

## 8. Availability Inputs

The `0.5` availability truth inputs are fixed to this set:

- included calendars
- event transparency
- attendee response / declined-state filtering policy
- canceled-event exclusion
- evaluated time window
- all-day event handling rule
- source account / calendar inclusion policy

Default interpretation:

- declined events do not block unless policy says otherwise
- all-day events are still governed by transparency
- one default included-calendar set exists per provider account, with workspace/global overrides

Explicitly deferred:

- work hours
- travel buffers
- transit-aware buffers
- focus heuristics
- soft holds
- task-derived availability
- inferred preparation/recovery windows

### Availability explain output

Availability explain should surface at least:

- result
- blocking intervals
- sources consulted
- filters applied
- why candidate time was accepted or rejected
- basis or source-of-truth class such as `exact | inferred | mixed`
- optional lightweight confidence such as `high | medium | low`

## 9. Tombstone Restore Rule

This rule is uniform across providers:

When an upstream object reappears and identity match is sufficiently strong, revive the existing canonical object and reconcile via the existing or historical `SyncLink`.

Do not create a brand-new canonical object unless identity cannot be safely re-established.

### Preserve on restore

- tombstone interval history
- linkage history
- audit lineage
- reconciliation explanation

### Restore posture

Only create a brand-new canonical object when restore confidence is too low or prior linkage is genuinely ambiguous/corrupt.

A disappearing and reappearing upstream object should not fragment canonical history into multiple Vel identities by default.

## 11. Dry-Run Law

Dry-run semantics are hard law in `0.5`.

### Dry-run permits

- runtime records may be created
- audit entries may be created and must be marked `dry_run`
- proposed canonical diffs and proposed write ops may be computed as ephemeral plan/output artifacts
- proposed diffs/write ops may be stored only as dry-run runtime or audit artifacts, not as canonical content mutations

### Dry-run forbids

- canonical content mutation
- irreversible external mutation
- executed provider writes
- projection/cache refreshes or other side effects outside explicit dry-run runtime/audit records

## 12. Module Activation Vs Invocation

This distinction is top-level architecture law:

Activation grants availability; invocation is a separate policy-governed execution event.

### Additional rule

Activated modules may:

- sync
- index
- expose tools
- contribute context

Invocation still requires:

- explicit user/action/workflow/tool path
- policy check
- runtime evidence

This distinction should be treated as packet-index and architecture-overview law, not as an implementation footnote.

## 15. Google Auth / Integration Pattern

The Google integration/auth pattern must be shaped as a shared provider-account substrate, not as module-specific auth islands.

### Required rules

- multiple Google accounts supported
- incremental scope grants supported
- explicit granted-scope inspection supported
- provider-account-scoped token lifecycle
- shared auth substrate across Calendar / Contacts / Gmail / Drive / Docs
- module capability checks derive from granted scopes rather than assumptions

### Capability posture

Shape now for explicit capability strings such as:

- `google.calendar.read`
- `google.calendar.write`
- `google.contacts.read`
- `google.gmail.read`
- `google.gmail.send`
- `google.drive.read`
- `google.drive.write`
- `google.docs.read`
- `google.docs.write`

### Ledger posture

Use a generic provider-scope ledger now, with Google-first capability values.

Missing scopes should degrade safely for reads where possible and deny writes.

### Governance rule

Possessing Google auth and scope does not imply permitted execution.

Policy still governs:

- background sync
- context inclusion
- explicit tool use
- outward writes

### Shared-substrate rule

Google provider auth is a shared provider-account substrate, not a module-specific auth island.

Future Google Contacts, Gmail, Drive, and Docs integrations are expected to consume the same provider-account and granted-scope base rather than inventing parallel auth systems.

Use separate modules over one Google provider account rather than one Google super-module as the primary conceptual model.

Apple Reminders and future routine-oriented task providers should also map into the same canonical `Task` model rather than creating a separate task species.

This is post-`0.5` provider expansion, but the canonical task model should remain shaped to support it.
