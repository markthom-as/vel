# Phase 98 Plan

## Objective

Land the truthful runtime/config/integration seams needed for the single-node MVP before final UI completion and closeout.

## Implementation slices

1. Core settings and onboarding truth
   Status: completed.
   Scope:
   - add a canonical Core settings boundary for:
     - user name
     - node name
     - onboarding readiness state
     - developer override for disabled composer gate
     - universal structured/freeform operator profile
   - persist the minimum-setup-complete state
   - expose the onboarding nudge trigger truthfully rather than as frontend-only logic

2. Provider configuration and routing truth
   Status: completed.
   Scope:
   - persist provider definitions for local `llama.ccp` and OpenAI
   - support OpenAI API-key and OAuth-backed setup
   - persist provider priority ordering
   - persist ask-first provider behavior
   - inject universal operator profile into provider requests through a backend-owned seam
   - preserve provider-specific escape hatches instead of flattening all providers into one fake-common API

3. Google and Todoist lifecycle truth
   Status: completed.
   Scope:
   - make connect/reconnect/edit status truthful in `System`
   - ensure Google calendar read/write state is visible and actionable
   - ensure Todoist read/create/edit/complete/reopen/tag/project/due-date capabilities are visible and actionable
   - keep provider/integration icon/status information aligned with the real runtime state

4. `Now` and current-day contract truth
   Status: completed.
   Scope:
   - keep current-day boundary logic backend-owned instead of frontend-local, using the current local-session rollover substrate while richer bedtime/end-of-day and sunrise signals remain deferred until the runtime has those signals
   - expose overdue-plus-today task truth needed by the web surface
   - make drag-to-`Next Up` durably commit a task to today through backend-owned state instead of UI-only behavior
   - keep Todoist as the effective source of truth while preserving Vel’s classification layer as a boundary-owned projection

5. Multimodal thread input truth
   Status: completed.
   Scope:
   - add or tighten the thread/runtime seams needed for image/file input
   - keep assistant/provider failures mappable to retry-capable nudges

6. Thread call-mode truth
   Status: completed.
   Scope:
   - persist a thread-owned `call_mode_active` flag on the canonical conversation seam
   - let the thread header start or end call mode through the existing conversation patch path
   - keep browser speech-to-text on the normal assistant-entry flow
   - speak assistant replies locally only when the active thread has call mode enabled
   - avoid widening into a separate realtime calling stack

## Verification

- focused Rust tests for changed service/storage/API seams
- targeted web boundary/data-layer tests where DTO/config shapes change
- direct runtime checks for onboarding gating, provider config, and integration mutation flows where cheap and available

## Success Criteria

1. Core settings, onboarding readiness, and provider/integration config are persisted truthfully rather than implied by UI state.
2. `Now` day semantics and drag-to-commit behavior stop depending on frontend-only approximation.
3. Multimodal input and call-mode support have real backend/runtime seams instead of placeholder UI affordances.
4. Phase 99 can focus on interaction completion and polish rather than missing truth contracts.
