# Phase 97 Plan

## Objective

Produce the execution checklist and lock decisions that Phase 98 and later phases must follow for `0.5.6`.

## Deliverables

- [ ] a locked MVP must-pass flow list
- [ ] a locked Core-settings/onboarding contract
- [ ] a locked provider/integration matrix
- [ ] a locked `Now` / `Threads` / `System` acceptance map
- [ ] a locked operator-mode vs developer-mode disclosure split
- [ ] a locked closeout method for manual desktop-Chrome QA

## Execution Checklist

### 1. Lock the MVP entry gate

- [ ] onboarding begins with a persistent onboarding nudge
- [ ] onboarding routes into a new Core settings area
- [ ] Core settings must support:
  - [ ] user name
  - [ ] node name
  - [ ] at least one LLM provider
  - [ ] at least one synced provider
  - [ ] structured operator profile fields
  - [ ] freeform operator profile notes
- [ ] floating composer/task bar stays disabled until minimum Core setup is complete
- [ ] developer mode includes an override for this gate

### 2. Lock provider and integration minimums

- [ ] required chat providers for milestone close are local `llama.ccp` and OpenAI
- [ ] OpenAI support includes API key and OAuth
- [ ] provider selection is controlled by a priority list
- [ ] provider settings can mark a provider as ask-first
- [ ] operator profile context is injected universally across providers
- [ ] Google support includes auth/connect, reconnect, settings edit, calendar read, calendar write, and `Now` event rendering
- [ ] Todoist support includes auth/connect, reconnect, settings edit, read/create/edit/complete/reopen, tags, projects, due dates, and `Now` task rendering

### 3. Lock `Now` behavioral truth

- [ ] today view includes all uncompleted Todoist tasks for today
- [ ] overdue tasks appear together with today tasks using explicit overdue styling
- [ ] Vel may maintain a local classification layer, but Todoist remains effective truth in this milestone
- [ ] bedtime/end-of-day defines the current-day window when present
- [ ] sunrise-next-day is the fallback boundary when no bedtime/end marker exists
- [ ] dragging a task to `Next Up` durably commits it to today

### 4. Lock `Threads` behavioral truth

- [ ] default thread state is last thread or no thread
- [ ] no-thread view has a deliberate empty state
- [ ] sending must not duplicate the user message
- [ ] thread layout uses modern left/right chat bubbles with no tails
- [ ] thread-level call mode supports speech-to-text plus text-to-speech over the normal assistant flow
- [ ] call mode may be deep-linked from a nudge

### 5. Lock `System` disclosure and control posture

- [ ] normal operator mode hides MVP-irrelevant technical/runtime internals
- [ ] developer mode reveals deeper technical settings and overrides
- [ ] unavailable services remain visible in a collapsed-by-default section
- [ ] integration/provider failures surface as nudges
- [ ] assistant/provider failures surface as nudges with retry when appropriate
- [ ] docs render inside the site frame

### 6. Lock acceptance method

- [ ] milestone acceptance authority is manual operator QA
- [ ] primary acceptance environment is desktop Chrome
- [ ] mobile web is not a primary acceptance target in `0.5.6`
- [ ] closeout audit must re-check the verbatim copied `TODO.md` feedback directly
- [ ] anything unfinished at close must be deferred explicitly with a reason

## Exit Criteria

- every item above is either locked in milestone docs or explicitly deferred
- later phases can execute without reopening MVP/product-definition questions
- the roadmap for Phases 98-100 stays implementation-first rather than clarification-first
