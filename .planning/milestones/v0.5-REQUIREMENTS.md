# Milestone v0.5 Requirements Archive

**Status:** SHIPPED 2026-03-23
**Milestone:** v0.5

## Outcome

All milestone-scoped `v0.5` requirements were satisfied. Remaining items noted below are accepted deferred work or intentionally narrow seams, not missed milestone commitments.

## Milestone Acceptance Checklist

- [x] canonical objects became the system of record
- [x] reads and writes flow through a typed action membrane with dry-run and audit evidence
- [x] ownership, grants, confirmation posture, and policy govern mutating behavior
- [x] core, Todoist, and Google Calendar modules load through one governed path
- [x] workflows and skills exist as governed backend primitives over canonical objects
- [x] calendar semantics are native in core instead of provider-defined
- [x] Todoist maps into canonical task semantics with multi-account sync and mediated writes
- [x] Google Calendar maps into canonical calendar semantics with multi-account sync and mediated writes
- [x] hard cutover retired legacy write authority and left only bounded compatibility seams
- [x] milestone proof is execution-backed

## Requirement Outcomes

### Core Substrate

- [x] **CORE-50-01**: canonical object kernel exists with typed envelopes, relations, schemas, and external linkage metadata
- [x] **CORE-50-02**: Vel becomes the system of record for canonical objects, not a thin mirror of provider data
- [x] **CORE-50-03**: multi-account integration identity, cursors, provenance, and ownership are built into the core substrate

### Action / Policy / Audit

- [x] **ACTION-50-01**: all reads and writes flow through a typed action membrane with stable schemas and dry-run support
- [x] **POLICY-50-01**: grants, field ownership, confirmation posture, and audit logs govern mutating behavior

### Module / Workflow Runtime

- [x] **MODULE-50-01**: core, Todoist, and Google Calendar modules load through one registry/loader path
- [x] **WF-50-01**: skills and workflows exist as first-class governed backend primitives with typed context binding

### Calendar / Task Adapters

- [x] **CAL-50-01**: canonical calendar objects cover calendars, events, attendees, recurrence, locations, and availability
- [x] **TODOIST-50-01**: Todoist tasks/projects/sections/labels/comments and sync state map into canonical Vel objects
- [x] **GCAL-50-01**: Google Calendar calendars/events/attendees/recurrence/location/availability map into canonical Vel objects
- [x] **SYNC-50-01**: bidirectional sync works with explicit ownership and source-favored conflict resolution on source-owned fields

### Cutover / Verification

- [x] **CUTOVER-50-01**: existing backend seams are retired and the new canonical core becomes the live backend path
- [x] **VERIFY-50-01**: milestone verification proves backend authority, adapter correctness, and policy/audit behavior with execution-backed evidence

## Accepted Deferred Work

- [ ] non-`0.5` provider write migrations for Notes, Apple Reminders, GitHub, and Email
- [ ] client/web/Apple rebuild against canonical `0.5` backend contracts
- [ ] scheduler, trigger, and background automation over the workflow runtime
- [ ] broader connector expansion beyond Todoist and Google Calendar
- [ ] richer availability policy, behavior analytics, and broader task/calendar semantics beyond the frozen `0.5` packet
