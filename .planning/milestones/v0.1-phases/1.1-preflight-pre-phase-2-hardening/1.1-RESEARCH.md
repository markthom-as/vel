## Summary

Phase 1.1 is purely technical debt elimination across three independent work items. Each item is bounded and self-contained. There are no new external libraries needed — all three items use patterns and APIs already present in the codebase or in the installed version of sqlx.

**Key finding on panics:** After exhaustive grep of the codebase, there are no `expect()` or `unwrap()` calls at production runtime in `integrations_todoist.rs` or `integrations_google.rs` themselves. The original audit concern was about `load_settings` in these files' shared `load_settings<T>` helper — which uses `serde_json::from_value::<T>(value).unwrap_or_default()` (safe, already handled). However, several **production-path panics** exist in the broader veld crate that are genuine crash risks: `adapters/calendar.rs:54`, `routes/runs.rs:440`, `routes/suggestions.rs:328`. The phase scope as specified targets the integration services; the planner should scope to the stated work items.

**Key finding on WAL mode:** sqlx 0.8 defines `SqliteJournalMode::Wal` as the default *enum variant*, but explicitly documents and implements that journal_mode is NOT applied unless the caller invokes `.journal_mode(...)`. The current `connect_pool` never calls this method. WAL mode is therefore not active. Enabling it requires exactly one line change plus an in-memory guard.

**Key finding on app.rs decomposition:** The auth-middleware surface in `app.rs` is entirely self-contained in lines 1–474 (all before the `#[cfg(test)]` block at line 476). The symbols to extract are: `RouteExposureClass`, `HttpExposurePolicy`, `ExposureGate`, `env_flag_enabled`, `extract_bearer_token`, `extract_header_token`, `expected_token_for_class`, `unauthorized_response`, `enforce_exposure_gate`. The remaining file (10,560 lines of tests) calls `HttpExposurePolicy` directly via `use super::*`, so after extraction the test module requires updating to `use crate::middleware::*` (or a re-export from `app.rs`).

**Primary recommendation:** Execute the three items in this order: (1) integration panic hardening (lowest risk, no API change), (2) WAL mode (single line, guarded), (3) app.rs decomposition (highest surface area due to 160 tests using `HttpExposurePolicy`).

## Area 2: SQLite WAL Mode

### Current State

**File:** `crates/vel-storage/src/infra.rs`, function `connect_pool` (lines 28-41)

```rust
pub(crate) async fn connect_pool(db_path: &str) -> Result<SqlitePool, StorageError> {
    if db_path != ":memory:" {
        if let Some(parent) = Path::new(db_path).parent() {
            fs::create_dir_all(parent)?;
        }
    }

    let pool = SqlitePoolOptions::new()
        .max_connections(1)
        .connect_with(sqlite_connect_options(db_path)?)
        .await?;

    Ok(pool)
}

pub(crate) fn sqlite_connect_options(db_path: &str) -> Result<SqliteConnectOptions, StorageError> {
    let url = if db_path == ":memory:" {
        "sqlite::memory:".to_string()
    } else if db_path.starts_with("sqlite:") {
        db_path.to_string()
    } else {
        format!("sqlite://{db_path}")
    };

    let options = SqliteConnectOptions::from_str(&url)?.create_if_missing(true);

    Ok(options)
}
```

**Problem:** `.journal_mode(SqliteJournalMode::Wal)` is never called. Despite `SqliteJournalMode::Wal` being the `#[default]` enum variant, sqlx 0.8 explicitly does NOT apply it unless the caller invokes `.journal_mode(...)`. See sqlx source `options/mod.rs` line 177: "Don't set `journal_mode` unless the user requested it."

**Current mode:** DELETE (SQLite default for file-based DBs) — single-writer, readers block during writes.

### sqlx 0.8 WAL API (HIGH confidence — verified from installed source)

```rust
// from sqlx-sqlite-0.8.6/src/options/mod.rs
use sqlx::sqlite::{SqliteConnectOptions, SqliteJournalMode};

let options = SqliteConnectOptions::from_str(url)?
    .create_if_missing(true)
    .journal_mode(SqliteJournalMode::Wal);
```

**Import path:** `use sqlx::sqlite::{SqliteConnectOptions, SqliteJournalMode};`
- `SqliteJournalMode` is pub-exported from `sqlx::sqlite` (confirmed in `lib.rs`)

### WAL + In-Memory Databases

SQLite documentation: WAL mode requires a real file (uses `-wal` and `-shm` sidecar files). In-memory databases use `MEMORY` journal mode by default and cannot use WAL.

**Guard required:** Apply WAL only for non-memory databases.

```rust
pub(crate) fn sqlite_connect_options(db_path: &str) -> Result<SqliteConnectOptions, StorageError> {
    let url = if db_path == ":memory:" {
        "sqlite::memory:".to_string()
    } else if db_path.starts_with("sqlite:") {
        db_path.to_string()
    } else {
        format!("sqlite://{db_path}")
    };

    let mut options = SqliteConnectOptions::from_str(&url)?.create_if_missing(true);

    if db_path != ":memory:" {
        options = options.journal_mode(SqliteJournalMode::Wal);
    }

    Ok(options)
}
```

### PRAGMA Ordering Note

The sqlx source (options/mod.rs line 167-175) documents the correct PRAGMA ordering:
1. `locking_mode` (before journal_mode)
2. `auto_vacuum` (before journal_mode)
3. `journal_mode` (last)

Calling `.journal_mode()` via sqlx handles ordering automatically via the HashMap-ordered pragma application. The `connect_with` method applies all pragmas in the order the HashMap preserves them. No manual PRAGMA sequencing is needed.

### Pool Size and WAL

Current pool: `max_connections(1)` — single connection. WAL still provides value:
- Readers never block on writes (WAL readers see the last committed snapshot)
- Background workers (tokio tasks) sharing the single connection via pooling benefit from WAL's non-blocking read model
- Phase 2 will increase concurrency pressure — WAL should be in place before then

The `max_connections(1)` setting is appropriate for SQLite and does not need to change as part of WAL enablement.

### Existing Tests That Will Be Affected

All tests using `Storage::connect(":memory:")` are unaffected (guard prevents WAL on memory).

Tests using real file paths (tmp files) in `services/integrations.rs` (lines 1370-1396) would use WAL after this change — acceptable, WAL is strictly more permissive.

No existing test asserts `journal_mode = DELETE` so no tests will break.

### New Tests Needed
- Test: `connect_pool` with a real file path → PRAGMA journal_mode returns "wal"
- Test: `connect_pool` with `:memory:` → does not error (WAL guard works)

Pattern for the file-based test:
```rust
// Use a temp file, connect, query PRAGMA journal_mode, assert = "wal"
let tmp = tempfile::NamedTempFile::new().unwrap();

```

Note: `tempfile` crate is not currently in Cargo.toml. Alternative: use a temp dir path with `std::env::temp_dir()` (pattern already used in `integrations.rs` test helpers at line 1356).

## Common Pitfalls

### Pitfall 1: WAL on In-Memory Databases
**What goes wrong:** Calling `.journal_mode(SqliteJournalMode::Wal)` on a `:memory:` database causes SQLite to silently ignore the pragma or return an error (WAL requires real file I/O for the WAL and SHM files).
**How to avoid:** Guard with `if db_path != ":memory:"` before applying WAL. The current `connect_pool` already has this guard pattern for `create_dir_all`.
**Warning signs:** Tests fail with "unable to open database file" or WAL pragma returns a mode other than "wal".

### Pitfall 2: sqlx WAL Default Enum vs. Applied Setting
**What goes wrong:** Developer sees `#[default] Wal` on `SqliteJournalMode` and concludes WAL is already enabled. It is not. sqlx explicitly sets the pragma map entry to `None` for `journal_mode` and only applies it when the caller calls `.journal_mode(...)`.
**How to avoid:** Always verify by querying `PRAGMA journal_mode` after connection in a test.

### Pitfall 3: app.rs Test Imports After Middleware Extraction
**What goes wrong:** Moving `HttpExposurePolicy` out of `app.rs` breaks the test module's `use super::*` pattern — the struct is no longer in scope.
**How to avoid:** Add `pub use crate::middleware::HttpExposurePolicy;` (and other used symbols) to `app.rs` before removing the original definitions, or update the test module's import block.

### Pitfall 4: HttpExposurePolicy Field Visibility After Move
**What goes wrong:** Fields of `HttpExposurePolicy` are currently implicitly `pub(super)` (private to `app.rs`). Tests in the same file can construct struct literals. After moving to `middleware/mod.rs`, the fields must be `pub(crate)` for test construction to work across module boundaries.
**How to avoid:** Add `pub(crate)` to each field when writing `middleware/mod.rs`.

### Pitfall 5: app.rs Line Count Expectation
**What goes wrong:** After extracting ~120 lines of auth logic, `app.rs` drops from 11,036 to ~10,916 lines — far above the "3,000 line" success criterion. This is because 10,560 lines are tests that cannot move.
**How to avoid:** The success criterion likely refers to the non-test production section (lines 1-474 → reduced to ~350 after extraction). Confirm interpretation. Alternatively, consider a follow-up task to split tests across per-feature test modules, but that is out of scope for this phase.

## Architecture Patterns

### Rust Module Extraction Pattern

```
Step 1: Create middleware/mod.rs with extracted symbols (pub(crate) visibility)
Step 2: Add re-exports in app.rs: pub use crate::middleware::{...};
Step 3: Verify tests compile: cargo test -p veld
Step 4: Remove original definitions from app.rs
Step 5: Remove re-exports (or keep for ergonomics)
Step 6: Update lib.rs to declare pub mod middleware;
```

Doing steps 1-3 before 4-6 allows incremental verification without breaking the build at any point.

### sqlx PRAGMA Application Pattern (sqlx 0.8)

```rust
// CORRECT: Use the typed API
let options = SqliteConnectOptions::from_str(&url)?
    .create_if_missing(true)
    .journal_mode(SqliteJournalMode::Wal);

// AVOID: Raw PRAGMA string
sqlx::query("PRAGMA journal_mode=WAL").execute(&pool).await?;
// ^ Works but bypasses sqlx's ordering guarantees and doesn't persist across pool connections
```

### Integration Settings Hardening Pattern

```rust
// Current (safe but silent on corruption):
.map(|value| serde_json::from_value::<T>(value).unwrap_or_default())

// Improved (safe + observable):
.map(|value| serde_json::from_value::<T>(value).unwrap_or_else(|err| {
    tracing::warn!(key = %key, error = %err,
        "integration settings deserialization failed, using defaults");
    T::default()
}))
```
