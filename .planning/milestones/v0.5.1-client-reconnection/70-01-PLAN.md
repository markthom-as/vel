# 70-01 Plan

## Objective

Reconnect `Threads` and introduce `/system` as the single authoritative structural surface.

## Work

1. Rebind `Threads` to canonical interaction truth and allow invocation only when exactly one stable bound canonical object with a supported workflow binding exists.
2. Show an explicit attach/create-object affordance for object-less threads rather than allowing floating invocation.
3. Ship `/system` as a single-route surface with stable internal sections for `Domain` (`People`, `Calendar`, `Knowledge`), `Capabilities` (`Tools`, `Workflows`, `Templates`), and `Configuration` (`Modules`, `Integrations`, `Accounts`, `Scopes`).
4. Expose invocation, explain, and state visibility only through canonical backend outputs.
5. Lock `Modules`, `Integrations`, `Accounts`, and `Scopes` to the pre-frozen canonical action allow-list:
   - `Modules`: `activate`, `deactivate`
   - `Integrations`: `connect`, `disconnect`, `refresh` only if already canonical in the backend
   - `Accounts`: `reauthorize`, `disconnect`
   - `Scopes`: `enable`, `disable` only if already canonical in the backend

## Acceptance

- `Threads` no longer depends on stale client semantics
- object-less and multi-object workflow invocation are impossible
- `/system` exists as one route, not a mini-app cluster
- no additional, inferred, composite, or ambiguous configuration actions are rendered
- `Templates` remain list/read-only unless a real canonical apply/invoke action already exists
- `Knowledge` remains thin browse-plus-stable-action territory only
- no builders/editors/simulators are introduced
