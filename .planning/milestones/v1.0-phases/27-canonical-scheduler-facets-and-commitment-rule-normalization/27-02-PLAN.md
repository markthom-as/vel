# 27-02 Plan

## Slice

Persist normalized scheduler facets for commitments and expose them through backend/domain seams.

## Why this slice second

The normalized rule model is only useful if it becomes durable and queryable. This slice gives reflow, recall, and assistant context a stable substrate to consume.

## Outcomes

- store canonical scheduler facets in durable Vel-backed form
- keep raw provider labels/tokens as compatibility metadata rather than durable semantics
- expose the normalized fields through the backend/domain surfaces that already own commitments and related context
- keep storage and DTO boundaries clean

## Likely Touch Points

- `crates/vel-core/src/`
- `crates/vel-storage/src/`
- `crates/veld/src/services/`
- `crates/vel-api-types/src/` if transport contracts widen

## Verification

- focused storage/domain tests for round-trip persistence
- migration or repo tests proving normalized facets survive load/save paths
- boundary tests showing raw labels are not the only source of truth after ingest
